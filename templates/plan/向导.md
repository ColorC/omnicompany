
# plan 模板填写指引

读这份的前提: 你已经看过 `范本.md` 知道一份合规 plan 长什么样, 现在要立一份新 plan。

## 起手两件事

### 命名迁移声明

新立的 plan 用 omnicompany 当前命名. 阶段拆分里描述工作时用"工人 / 团队 / 工作区 / 部门" (新名), 不用"router / pipeline / eventbus / project"等旧名. 完整对照在 `docs/standards/_global/terminology.md`. 旧 plan 已经写过的不强制改, 但新 plan 不传染旧名.

### 设施统一计划唯一入口

如果 plan 目标是消灭二重权威、统一核心设施或允许 agent 长程独立执行, 同目录必须补 `authority-confirmation.md` 与 `autonomous-execution-rules.md`。当前样例是 `docs/plans/agent-framework/[2026-06-13]LLM-CALL-UNIFICATION/authority-confirmation.md` / `autonomous-execution-rules.md`; 分散 plan 子文档只链接它们, 不复制第二套决断。

### plan 头部必有的 binding 块

按 `docs/standards/concepts/plan.md` 第三节, plan 头部 OmniMark v3 之后 / §1 主题之前必有一个 yaml-style binding 块:

```yaml
---
binding:
  workspace: <项目相对路径或绝对路径, 必填单值>
  packages:
    - service:<name>     # 0-N 个, 用前缀区分类型
    - domain:<name>
    - core:<name>
    - protocol:<name>
  targets:
    - team:<name>        # 0-N 个, 用 dashboard wikilink 已支持的 entity 前缀
    - worker:<name>
    - material:<id>
    - agent:<name>
    - hook:<name>
    - tool:<name>
    - plan:<topic>
applicable_standards:
  - standards/_global/code.md           # 所有 plan 都自动加这两条
  - standards/_global/llm_first.md
  - <按 targets / packages 推导的 standards/concepts/<kind>.md / standards/cli/<x>.md / 等>
expected_completion: <YYYY-MM-DD, 跟 §2 起止日字段一致>
ttl_days: <整数, 30-90 推荐, 长跨度的拆 plan>
---
```

字段填法跟漏填后果详见 plan 规范第三节, 这里讲下每字段的填空指引:

#### 字段: binding.workspace (必填单值)

干嘛: plan 实施过程中所有写盘动作的隔离边界. 跑代码 / 落数据 / 改文件都在这个 workspace 内.
怎么填:
- 项目相对路径: `omnicompany/` (整个 omnicompany 项目根, 适合跨服务大改)
- 子目录: `omnicompany/src/omnicompany/packages/services/<service>/` (单服务包改动)
- 独立 workspace: `omnicompany/data/_workspaces/<team>/<job>/` (一次 team 跑独立工作区)
漏填后果: 跑代码时不知道工作区边界, 跨 plan 互相污染, 守护扫到罚单. 违反 workspace_isolation_and_test_discrimination 第一条铁律.
检查项:
- [ ] workspace 路径真存在 (相对路径在项目里能 ls 到)?
- [ ] 单值不是数组 (一份 plan 一个 workspace)?

#### 字段: binding.packages (0-N 个, 推荐有)

干嘛: plan 触及的 package 列表, 让 plan 自动应用对应规范.
怎么填:
- 用前缀区分: `service:<name>` / `domain:<name>` / `core:<name>` / `protocol:<name>`
- 单服务包改动: `service:guardian` 一条
- 跨服务: 多条都列
- 业务领域: `domain:gameplay_system`
- 核心层: `core:cli` 跟 `core:bus` 等
- 跨业务跨核心: 多条都列
漏填后果: 守护不知道这份 plan 该按哪些规范审, 漏校验.
检查项:
- [ ] 列的 package 都真存在 (能在 src 里找到对应目录)?
- [ ] 前缀正确 (service / domain / core / protocol 之一)?

#### 字段: binding.targets (0-N 个, 推荐有)

干嘛: plan 直接动到的具体实体, 让 dashboard 自动生成跳转链接.
怎么填:
- 用 dashboard wikilink 已支持的 entity 前缀: `team:<name>` / `worker:<name>` / `material:<id>` / `agent:<name>` / `hook:<name>` / `tool:<name>` / `plan:<topic>`
- 跟 §4 关联材料.输出 字段对齐 (头部声明 + 正文铺开)
漏填后果: dashboard 不知道这份 plan 关心哪些实体, 跳转链接缺失, 实体改了不知道是哪份 plan 在动.
检查项:
- [ ] 跟 §4 关联材料.输出 一致?
- [ ] 用 wikilink 已支持的 entity 前缀, 不是自造?

#### 字段: applicable_standards (0-N 个, 自动 + 手补)

干嘛: 自动应用的 standards 列表, 守护跟 lint 引擎按这个列表选校验规则.
怎么填:
- 任何 plan 自动含: `standards/_global/code.md` + `standards/_global/llm_first.md`
- 按 targets 类型自动加: 含 `material:*` 加 `standards/concepts/material.md`, 同上对 worker / team / agent / hook / tool / data / template
- 按 packages 类型自动加: `core:cli` 加 `standards/cli/<x>.md` 等
- 调用方手动补: 例如涉及多份 DESIGN 时手补 `standards/protocol/design_md_template.md`
漏填后果: lint 引擎不知道按哪些规范查, 漏审.
检查项:
- [ ] 至少含两条全局 (`_global/code.md` + `_global/llm_first.md`)?
- [ ] 按 targets / packages 类型推导的 standards 都列了?

#### 字段: expected_completion (必填, ISO 8601)

干嘛: 预期完成日, 跟 §2 起止日字段值一致 (冗余但显式让 dashboard 直接读).
怎么填: ISO 8601 (YYYY-MM-DD).
漏填后果: ttl_days 算不准, dashboard 显示不出预期完成日.
检查项: [ ] 跟 §2 起止段写的预期结束日一致?

#### 字段: ttl_days (必填, 整数)

干嘛: 超期未收尾的报警阈值. 守护扫 aging / expiring_soon / expired 三档.
怎么填:
- 短期 plan (≤ 1 个月): 30
- 中期 plan (1-2 个月): 60
- 长期 plan (2-3 个月): 90
- 超过 3 个月: 不允许, 是项目不是 plan, 拆成多份
漏填后果: 守护扫不到提醒, plan 永远 active 沉淀债务.
检查项:
- [ ] 在 30-90 区间?
- [ ] 跟 §2 起止段跨度匹配 (跨度长 ttl 长)?

### plan 跟 compact_summary 的关系

plan.md 跟 compact_summary_<date>.md 是配对存在的两份材料:

- **plan.md** (一份): 跨日跨阶段的过程记录, 计划本身的当前状态. 阶段进展时回头改这份, 不再每次新建.
- **compact_summary_<date>.md** (多份, 按日期累加): 每次 trigger 时机产生的快照 — compact 前 / 用户要求总结 / 开新 team / 验证收尾, 任一即触发. 多次 compact 累加多份 compact_summary, 不覆盖。

两份的关系是 plan 是常规进度记录, compact_summary 是 trigger 时机的快照。compact_summary 引用 plan 的当前状态 + 加 §9-§13 几节假设 / 验证 / 实验 / 假设树 / 一句话总览。

普通 plan 编辑时填 §1-§8 即可, §9-§13 留空。compact_summary 时把整份 plan 复制成 `compact_summary_<date>.md`, 然后在 §9-§13 填上当时的假设 / 验证 / 实验等内容。完整规程在 `docs/standards/protocol/l2_session_summary_protocol.md`。

### 第一件: 这件事是不是该立 plan?

立 plan 之前自问:

- 这件事跨度多长? 一天能做完 → 不立 plan, 直接做; 跨数日到数周 → 立 plan
- 涉不涉及多个阶段? 单步骤 → 不立; 三个阶段以上 → 立
- 需不需要用户审议关键决策点? 全自决 → 可不立; 需要用户审 → 立

不该立 plan 的事 (例如修一个小 bug / 改一份文档 / 跑一次自检) 强行立 plan 是浪费 — 让 plan 系统沉淀垃圾。

### 第二件: 决定主题代号 + 起始日期

主题代号大写蛇形, 5-30 字符, 描述这份计划在做什么。例子:
- `OMNICOMPANY-CLI-PHASE3` (好 — 短, 说明阶段)
- `TEAM-BUILDER-RECONSTRUCT` (好 — 动词在前)
- `2026-05-01-cli-impl` (不好 — 不该带日期, 日期已经在目录前缀里)
- `cli` (不好 — 太短, 说不清)

起始日期用 ISO 8601 (YYYY-MM-DD), 是计划真正开始动的那天 (不是想起来的那天)。

## 字段: OmniMark 头

干嘛: 让这份 plan 在文档系统里有可追溯身份。
怎么填:
- `origin` 通常 `ai-ide` (AI IDE 写的) 或 `human` (用户手写)
- `domain` 写 `docs/plans/<主题代号>` 跟目录名对齐
- `type` 必须 `plan`
- `summary` 一句话讲计划做什么
- `why` 一句话讲起因 / 跟主轴关系
- `tags` 至少含 `plan` + 主题领域 + 阶段标签
漏填后果: 守护扫到没头的 plan 标违规, 文档系统找不到归属。
检查项:
- [ ] is_canonical_v3 通过
- [ ] type 是 `plan`

## 字段: 起源段 (quote)

干嘛: 在主标题之后用 `> 起源:` 引用块写起因, 比 OmniMark why 字段详细。
怎么填:
- 2-3 句话, 讲"为什么这一刻要立这份 plan"
- 包括: 上一阶段做完了什么 / 当前发现什么问题 / 这份 plan 怎么解决
- 例: "起源: 2026-04-15 设施层第二轮收官. template / data / sandbox 都立了, 但注册中心还没真做出来 — 没有它, 上层模板系统是'规范立了但不能用'的状态. 阶段三集中实装注册中心。"
漏填后果: 后续读这份 plan 的人不知道为什么立的, plan 跟主轴脱钩。
检查项:
- [ ] 起源段 2-3 句话, 不超过半屏?
- [ ] 一个外部审阅者读完起源段能复述这份 plan 的起因?

## 字段: §1 主题

干嘛: 让所有读这份 plan 的人 30 秒内知道这份计划要做什么。
怎么填:
- 2-3 句话, 跨越范围 + 大致目标
- 不堆代号, 中文铺开
- 不要复述 OmniMark summary 字段, 要展开
漏填或敷衍后果: 后续团队接手时不知道这份 plan 为什么存在, 容易跟其他 plan 重复。
检查项:
- [ ] 跟跨越范围一致 (不是讲某个具体阶段)?
- [ ] 不堆代号?

## 字段: §2 起止日期

干嘛: 给 plan 时间预算和度量基准。
怎么填:
- 起始日期: ISO 8601 格式, 跟目录名前缀一致
- 结束日期: 预期完成日 (允许后续修订)
- 括号里写估计天数 + 谁 (全职/兼职)
- 估计天数有依据 — 应当是 §5 阶段拆分加总, 不是拍脑袋
漏填后果: 没有预算就没有"逾期"概念, 计划永远在跑, 最后变成"项目"而不是"计划"。
检查项:
- [ ] 起始日期跟目录名一致?
- [ ] 估计天数 = §5 阶段拆分加总?
- [ ] 跨度 ≤ 3 个月? 超过应当拆成多份 plan 或者立 project (如果以后有 project kind)

## 字段: §3 参与方

干嘛: 让审议环节知道找谁; 让重叠工作的人知道彼此在做什么。
怎么填:
- 主实施者明确写清楚 (谁负责写代码 / 跑测试 / 写文档)
- 审议者: 哪些环节需要谁审 (用户 / 其他工人 / 守护)
- 其他参与方: 例如守护作 sidecar (实装期间持续运行验证)
漏填后果: 审议时找不到决策人, 决策没人拍板。
检查项:
- [ ] 主实施者明确?
- [ ] 用户 / 守护 等关键审议方列入了 (如适用)?

## 字段: §4 关联材料

干嘛: 让这份 plan 在材料图里有位置, 不悬空。
怎么填:
- 上游 (基于哪些已有材料): 列材料 ID 或路径, 至少 1 项
- 输出 (会产出哪些新材料): 列预期产出, 至少 1 项
- 关联材料 ID 真存在 (能在注册中心或文档系统查到)
漏填后果: plan 做完后产出材料没有 traceback 到这份 plan, 注册中心找不到归属。
检查项:
- [ ] 上游 ≥ 1 项 + 输出 ≥ 1 项?
- [ ] 上游材料路径都能打开?

## 字段: §5 阶段拆分

干嘛: 把"起止日期"那个总预算分到具体步骤上, 让进展可追踪。
怎么填:
- 编号阶段, 每条带阶段名 + 估计天数
- 3-7 个阶段比较合理 (太多 = 颗粒度太细看不出主线, 太少 = 拆分不够中途失败时不知道炸哪)
- 每阶段产出物明确 (不是"做完那阶段")
- 注明每阶段是不是需要用户审议 (一般每阶段都审, 除非完全自决的小步)
漏填后果: 计划看着是一整块, 中途失败时不知道哪一段炸了, 不知道还剩多少。
检查项:
- [ ] 阶段总天数加起来 = §2 估计天数?
- [ ] 每阶段产出明确?
- [ ] 哪些阶段需要用户审议标了出来?

## 字段: §6 收尾条件

干嘛: 定义"什么算这份计划完成"的硬标准, 防止计划无限拖。
怎么填:
- 一两条具体可测的条件 (例如 "X 测试通过" + "Y 文档落盘")
- 跟 §4 输出材料对齐 — 输出都齐 + 通过验证 = 收尾
- **不能**是"差不多就行" / "感觉做完了" 这种模糊措辞
漏填或填模糊后果: 计划没有终止线, 被无限"还有一点点要补"拉长。
检查项:
- [ ] 收尾条件可测 (有客观标准)?
- [ ] 跟 §4 输出对齐?

## 字段: §7 风险

干嘛: 提前说出已识别的不确定性, 让审议者能挑战。
怎么填:
- 2-5 条已识别风险, 每条简短描述发生时的影响 + 缓解办法
- 风险跟当前阶段实际情况贴合 — 不抄历史模板
漏填后果: 风险隐而不发, 撞上时反应慢。
检查项:
- [ ] 2-5 条?
- [ ] 每条都有缓解办法 (不是只说有风险但没对策)?

## 字段: §8 收尾归档位置

干嘛: 计划做完后产出物归档到哪。
怎么填:
- 按 `docs/standards/_global/distributed-docs.md` §5.3 主题区单轴结构, 落到下列三类之一的顶层目录:
  - **改具体代码模块** → `docs/plans/<代码模块名>/[<起始日期>]<主题代号>/` (例 `agent-framework` / `dashboard` / `diagnosis` / `format-material` / `guardian`)
  - **服务具体业务域** → `docs/plans/<业务域名>/[<起始日期>]<主题代号>/` (例 `voxel_engine` / `gameplay_system`)
  - **跨切面方法论 / 元能力 (行政层)** → `docs/plans/omnicompany-<能力名>/[<起始日期>]<主题代号>/` (例 `omnicompany-计划跟进` / `omnicompany-调研吸收`)
- 判定不出时优先级: 代码模块 > 业务域 > 行政层 (能落到代码就落到代码, 不强行抽象)
- 跟 §2 起始日期 + 主题代号一致
- 完成 + 冷静期 (2-4 周) 过后, 移到本主题区的 `_archive/[<起始日期>]<主题代号>/`
- **不允许**走旧的 `docs/plans/_infra/` `docs/plans/_cross/` `docs/plans/domain/` `docs/plans/_projects/` `docs/plans/_capabilities/` 路径 (均已废)

漏填或路径不对后果: 守护扫到不合规路径报警, 后续找历史困难.

检查项:
- [ ] 顶层是真实的代码模块名 / 业务域名 / `omnicompany-<能力名>` 三类之一?
- [ ] 起始日期跟 §2 一致?
- [ ] 没用废弃路径 (`_infra/` `_cross/` `domain/` `_projects/` `_capabilities/`)?

## 字段: §14 债务清单 (收尾必填)

干嘛: 实施过程中产生的债务列档, 防止 plan 收尾时遗留没人接手的开放项.
怎么填:
- 表格形式, 每条带: 债务描述 / 类型 (代码/文档/测试/配置) / 严重度 (blocker/major/minor) / 处置
- 处置必须三选一: 已解决 / 新立接手 plan (用 `[[plan:NEW-TOPIC]]` 链接) / 接受为永久债务 (并归档到 docs/debt/)
- 不允许"待补"开放项 — status 改 completed 之前每项必有处置
漏填或留待补后果: plan 收尾后债务遗留, 接手的人重复踩坑, 跟"防债务"目标背离.
检查项:
- [ ] 收尾时表格非空 (实施过程总会有些债务, 真无任何债务的概率极低)?
- [ ] 每项有处置, 没"待补"?
- [ ] 新立接手 plan 的用 `[[plan:NEW-TOPIC]]` wikilink 链接了?

## 字段: §9 工作性假设 (compact summary 时机才必填)

干嘛: 显化决策依据 — 每个 worker / team / 阶段背后的工作性假设, 让接手的人知道当时在赌什么。
怎么填:
- 每个 worker 至少对应 1 条假设, 形如 "假设 [X 内容] 可以在 [Y 情况] 下工作"
- 每个 team 至少对应 1 条假设, 形如 "假设 [X SOP] 可以在 [Z 信息空间] 内处理好"
- 假设要写在哪里: 优先放对应 service 的 DESIGN.md, compact summary 时复述到这里
- 普通 plan 编辑时这一节可省, 留空
漏填后果 (compact summary 时机): 决策没显化, 接手的人重头猜, 跟 omnicompany "数据密集型决策显化" 的目标背离。
检查项 (compact summary 时机):
- [ ] 每个 worker / team / 阶段都有对应假设?
- [ ] 假设格式是"假设 X 在 Y 情况下工作", 不是模糊措辞?

## 字段: §10 验证情况 + 验证背后的假设 (compact summary 时机才必填)

干嘛: 列出关键产物的验证状态 + 每个验证背后的必要性假设。
怎么填:
- 表格列产物 / 是否验证 / 验证方法 / 结果 (PASS / FAIL / 无验证)
- 每个验证背后的假设形如 "如果 [X 健康], 那 [Y 方面] 应当表现为 [Z]"
- 没验证的产物显式记录, 优先级低于数据库, 但出问题时必补阶段性验证
漏填后果: 验证孤立存在, 不知道验证是基于什么假设设计的, 假设本身错了验证白做。
检查项 (compact summary 时机):

## 字段: §11 假设树 (compact summary 时机才必填)

## 字段: §12 假设验证情况 + 一句话总览 (compact summary 时机才必填)
