
# 新建一种 kind 模板的向导

读这份的前提: 你已经看过 [`范本.md`](范本.md) 知道一份完整模板长什么样, 现在要给某种新 kind (worker / agent / plan / 任何概念) 真做出四件套。

每一节对应骨架.md 里的一段填空, 教你怎么填。

## 起手: 先调研已有实现

**第零件事**: 在动手新建模板之前, 先扫现存代码里有没有同类实现。

若新模板会触及材料、LLM、agent、EventBus、guardian 这批核心设施统一概念, 先读 `docs/plans/agent-framework/[2026-06-13]LLM-CALL-UNIFICATION/authority-confirmation.md` 与 `autonomous-execution-rules.md`。模板只能把"唯一入口"写成锚点, 不允许复制另一套权威表。

具体做法:
- `grep -r '<相关概念关键词>' src/omnicompany/packages/` 找跟新 kind 相似的现存材料/工人/团队
- `grep -r 'class .*<新 kind>' src/omnicompany/` 找已经有的同名类
- 看 `omnicompany/docs/standards/_meta/standards-index.yaml` 里现有 kind 列表, 确认这个新 kind 不是已有概念的别名

如果发现已有同类实现:
- 已有合格的 → 用已有的, 不重复立新 kind, 模板这件事不用做
- 已有但质量不够 → 优先改进已有的, 不另起新 kind, 跟同事 / 用户对齐这是改还是另立
- 真完全是新概念, 系统里找不到任何同类 → 进下一步

跳过"调研已有"这一步直接立新模板的后果: 系统里出现重复实现 / 概念别名漂移 / 守护扫描时不知道哪份是权威。

## 起手: 先确认规范是否存在

**第一件事**: 先看 `omnicompany/docs/standards/<新 kind>.md` 是否存在。

- 不存在 → **不要先做模板**。先补规范。规范说"这种 kind 应该满足什么", 模板按规范产出"长什么样的就对", 没规范的模板跑出来无判断依据。
- 存在 → 读完规范再开模板。模板的"反模式"清单要从规范的反模式表抽取, 不能跟规范矛盾。

如果你想新立一种概念但还不确定应不应该有规范, 先在沙盒 `drafts/` 写规范草稿, 跟用户审议过, 立到 `docs/standards/<新 kind>.md`, 然后才回头做模板。

## 起手: 先确认跟现有概念的边界 — DESIGN.md / plan / 模板各管什么

**第二件事**: 看新 kind 跟其他形态文档的关系, 不要把它们混。

| 概念 | 长什么样 | 放哪 | 写谁的什么 |
|---|---|---|---|
| 模板 (本节讨论的) | 范本+骨架+向导+注册件四件套 | `omnicompany/templates/<kind>/` | 这种 kind 的实例**应该长什么样**, 跨业务通用 |
| DESIGN.md | 七节硬结构 | `src/omnicompany/packages/<服务>/DESIGN.md` | 某个具体服务包的设计是什么样, 服务级而非概念级 |
| plan.md | 8 节固定结构 | `docs/plans/[YYYY-MM-DD]TOPIC/plan.md` | 某次跨日跨阶段过程记录的当前状态, 计划级而非永久级 |

新建的是哪个层级?
- 永久存在的概念约束 (例如"工人应当满足什么") → 模板 + 规范
- 某个具体服务包的设计 → DESIGN.md, 不动模板
- 一次具体的工作过程 (例如"实施工人模板的修订") → plan.md, 不动模板

如果搞错层级, 模板会被滥用成 plan 的容器, 或者反过来 plan 被滥用成模板的具体内容, 两边都失效。

## 文件 1 / 4: 怎么写新 kind 的 `范本.md`

### 字段: OmniMark 头

干嘛: 给这份范本一个可追溯的身份, 让守护看到时知道是谁在什么时候为什么写的。
怎么填:
- `origin` 通常 `ai-ide`, 用户手写也可以是 `human`
- `domain` 写 `omnicompany/templates/<新 kind>`
- `ts` 当前时间 (ISO8601 UTC)
- `type` 必须是 `template` (不是新 kind 自己的类型 — 范本本身是模板的一部分)
- `summary` 一句话讲"这是 <新 kind> 模板范本"
- `why` 讲为什么这种 kind 需要专门的模板, 跟现有 kind 的边界
- `tags` 至少含 `template`, `<新 kind>`, `sample`

漏填后果: 守护扫到没头的范本会罚单化, 整份模板系统不合规。
检查项:
- [ ] 跑 `is_canonical_v3()` 返回 True
- [ ] type 是 `template`, 不是 `<新 kind>`

### 字段: 范本主体

干嘛: 让看的人不用查规范就能立刻知道"这种 kind 实例长什么样"。
怎么填:
- 选这种 kind 在系统里**真会出现**的形态作具体例子, 不要 foo/bar
- 字段全部填好, 没有任何 placeholder 残留
- 如果有显著子变体, 一档一份范本, 不混
- 范本里的具体值要跟现有代码风格一致, 不带过期 import 或废弃 API
漏填后果:
- 范本字段不全, 看的人不知道哪些是必有的
- 范本用占位词, 看的人不知道字段的语义
- 范本过期, 实例化出来从一出生就是历史包袱
检查项:
- [ ] 一个外部审阅者只看范本能复述这种 kind 的核心结构吗?
- [ ] 范本里的 import / 引用都是当前活的 (不指向 deprecated 模块)?
- [ ] 跟 `omnicompany/docs/standards/<新 kind>.md` 的字段约束完全一致?

## 文件 2 / 4: 怎么写新 kind 的 `骨架.md`

### 字段: OmniMark 头

干嘛: 跟范本.md 的头同语义。
怎么填: 跟范本.md 同, 但 `summary` 是"<新 kind> 模板骨架"。
检查项: 同范本.md 头检查项。

### 字段: 骨架字段集

干嘛: 给将来实例化时复制粘贴用的空填表。
怎么填:
- 字段位置跟范本.md 一一对应, 顺序一致, 数量一致
- 每个 placeholder 用 `<...>` 包起, 内容是**这个字段的语义说明**, 例如 `<2-3 句话讲这个工人是干嘛的>` 或 `<这个工人接收的输入材料 ID, 必须在材料注册中心查得到>`
- 不写 `<string>` `<int>` 这种类型描述 — 类型由规范规定, 骨架的 placeholder 给的是**语义提示**
- 不留可选字段 — 骨架里所有字段都是必填; 实例化时所有 placeholder 都要替换。如果某字段在某些实例可选, 让 placeholder 写 `<可选: 如果有 ... 就填, 没有填"无">`
漏填后果:
- 字段位置跟范本不一致 → 实例化时使用者按骨架填出来的实例字段顺序乱
- placeholder 是类型不是语义 → 使用者不知道字段的语义角色, 容易填错
- 留可选字段 → 实例化出来的实例字段不齐, 注册中心校验过不去
检查项:
- [ ] 字段顺序跟范本.md 一致 (逐字段对照)?
- [ ] 每个 placeholder 是语义提示不是类型?
- [ ] 没有"可选"字段悬空?

## 文件 3 / 4: 怎么写新 kind 的 `向导.md`

### 字段: OmniMark 头

干嘛: 跟范本/骨架同。
怎么填: 跟范本.md 同, `summary` 是"<新 kind> 模板填写指引"。
检查项: 同。

### 字段: 一字段一节

干嘛: 让填骨架.md 的人逐字段知道怎么填。
怎么填:
- 骨架.md 里每个字段都对应向导一节
- 一节回答四问题: **干嘛 / 怎么填 / 漏填后果 / 检查项**
- 每节内容紧凑, 不超过 1/4 屏 (经验值: 单字段说明不超过 200 字)
- 检查项是可勾选的硬条件, 不是模糊建议
漏填后果:
- 字段没对应向导节 → 使用者不知道怎么填这个字段
- 节展开成手册 → 使用者读到一半失去耐心, 不再认真看
- 检查项是模糊建议 (例如"内容质量好") → 使用者不知道达标判定
检查项:
- [ ] 骨架.md 的每个字段都有对应一节向导?
- [ ] 每节四问题都答?
- [ ] 检查项是可勾选硬条件?

### 字段: 整体检查项

干嘛: 填完所有字段后, 让使用者再走一遍发现遗漏。
怎么填: 5-10 个 checkbox, 包括 placeholder 残留检查、OmniMark 头检查、跟规范的关键对接点检查、文件位置检查。
漏填后果: 使用者填完单字段都过, 但整体可能有漏 (例如位置不对 / 头不齐 / 跟规范不一致)。
检查项:
- [ ] 整体检查项至少 5 条?
- [ ] 含 placeholder 残留检查?
- [ ] 含 OmniMark v3 五字段齐检查?
- [ ] 含跟 `docs/standards/<新 kind>.md` 至少一条硬条款的检查?

## 文件 4 / 4: 怎么写新 kind 的 `注册件.yaml`

### 字段: OmniMark 头

干嘛: 同。
怎么填: 用 `# [OMNI]` 不是 `<!-- [OMNI] -->` (因为是 yaml 文件)。`summary` 是"<新 kind> 模板的 CLI 注册件接口"。
检查项: 同。

### 字段: kind

干嘛: 标记这份注册件是给哪种 kind 用的。
怎么填: 跟 `<新 kind>` 字面一致 (例如 `kind: plan`)。
漏填后果: CLI 找不到对应注册件, 实例化拒。
检查项: [ ] 跟目录名 `omnicompany/templates/<新 kind>/` 一致?

### 字段: instance_naming

干嘛: 告诉 CLI 实例化时给文件起什么名。
怎么填:
- `pattern` 是文件名规则, 用 glob 风格 (例如 `*_worker.py`) 或固定名 (例如 `plan.md`)
- `description` 解释这种命名规则的语义理由
漏填后果: CLI 实例化时不知道叫什么, 走默认或者报错。
检查项:
- [ ] pattern 跟规范文件里的命名规则段一致?
- [ ] description 解释清楚为什么是这个规则?

### 字段: instance_location

干嘛: 告诉 CLI 实例化时把文件写到哪个目录。
怎么填:
- `pattern` 是项目相对路径, 用 `<var>` 形式让 CLI 在实例化时替换
- `variables` 给每个 var 解释怎么生成
- 路径必须在合法目录范围 (按规范定义), 不能写到沙盒 / 罚单 / 注册中心数据这些设施目录
漏填后果: 实例化时位置乱, 守护扫到位置不对的实例报警。
检查项:
- [ ] pattern 含至少一个 var 让 CLI 替换 (除非这种 kind 全部实例都在固定单一位置)?
- [ ] 每个 var 都有 description 解释生成规则?
- [ ] 跟规范文件里的位置段一致?

### 字段: required_fields

干嘛: 列实例必须有的字段, 给 CLI 校验用。
怎么填:
- `omnimark_header`: 列必填头字段, 至少有 origin/ts/type/summary/why/tags 六项
- `body_sections`: 如果实例是 markdown 类型, 列必有的章节标题 (按字面对应)
- `body_fields`: 如果实例是代码或 yaml 类型, 列必有的代码字段或 yaml key
漏填后果: CLI 跑 sandbox check 时不知道查什么, 实例化出来字段缺也不报。
检查项:
- [ ] omnimark_header 至少含五字段?
- [ ] body_sections 或 body_fields 至少有一项 (取决于实例类型)?
- [ ] 跟范本.md 的实际结构对得上?

### 字段: checks

干嘛: 给 CLI lint 引擎的可执行检查规则列表。
怎么填:
- 每条 check 含 `id` (kind-NNN 格式) + `description` (查什么) + `rule` (可执行判定逻辑)
- rule 表达式格式跟 omnicompany CLI lint 引擎对接 (具体 DSL 跟 CLI 实装阶段确定; 当前用伪代码, 实装时按需调整)
- checks 数量跟规范的硬条款对齐, 一条规范硬条款至少一条 check
漏填后果: 实例化产出的实例可能违反规范但 CLI 不报。
检查项:
- [ ] 至少 5 条 checks (覆盖头/位置/必填/规范关键约束)?
- [ ] 每条 description 清楚?
- [ ] 跟规范文件里的硬条款一一对应?

### 字段: header_defaults

干嘛: 给 CLI 实例化时自动填的 OmniMark 头默认值。
怎么填:
- `origin` 通常 `ai-ide` (调用方显式传可覆盖)
- `type` 必须 `<新 kind>` (不可覆盖)
- `status` 通常 `active`
- `agent` 用 `<auto-from-session>` 让 CLI 调 `current_writer_identity()` 拿
漏填后果: 实例化出来头字段空, is_canonical_v3 不过。
检查项:
- [ ] type 不可覆盖 (字面写 `<新 kind>`)?
- [ ] agent 用 auto 形式?

### 字段: related_standards

干嘛: 链接对应规范文件, 让 CLI / 守护回查时找得到。
怎么填: 列 `docs/standards/<新 kind>.md` 路径; 如果有相关其他规范也列。
漏填后果: 模板修订时不知道要同时改哪份规范, 容易脱钩。
检查项: [ ] 至少链接 `docs/standards/<新 kind>.md`?

### 字段: reference_anti_patterns

干嘛: 列这种 kind 常见的反模式, 让向导和实例化时引用作"千万别这样"清单。
怎么填: 2-5 条具体反模式, 从规范的反模式表抽取。
漏填后果: 反模式没沉淀进模板系统, 历史犯过的错容易再犯。
检查项: [ ] 跟规范的反模式表一致 (不矛盾)?

### 字段: counterexample_ledger

干嘛: 让这种 kind 实例化产出的实例里, 任何死规则触发都自动走反例账本。死规则在线监测的安全网。
怎么填:
- `required_for_implementations: true` (必有, 不允许 false)
- `ledger_path_pattern`: 反例账本落盘的路径模式, 例如 `data/services/<service>/counterexamples/<rule_id>.jsonl`
- `reflect_on_count: 1` (默认 1 条反例就触发反思, 不堆积)
- `degrade_on_count: 3` (默认 3 条反例触发规则降级或回退)
- `rule_trace_required_fields`: 反例账本落账时必有的字段集 (rule_id / rule_version / trigger_context / rule_output / timestamp)

漏填后果: 这种 kind 的实例如果含死规则 (硬编码 if / 分类 / 正则 / yaml 映射), 死规则触发时没有反例监测, 错了没人知道, 跟规范"立规则前必 LLM 验证 0 反例"的根本铁律脱钩。
检查项:
- [ ] required_for_implementations 是 true?
- [ ] ledger_path_pattern 含 `<rule_id>` 占位让按规则分文件?
- [ ] reflect_on_count 设了 (默认 1, 不要 ≥ 3 否则发现晚)?
- [ ] rule_trace_required_fields 至少含 rule_id / rule_version / trigger_context / rule_output / timestamp 五项?

### 字段: invariants

干嘛: 升级 test_baseline 让验证有判别力。光列绿/红样本路径是浅层基线, 真正的判别力靠不变量结构。
怎么填: 列每条不变量, 每条含:
- `invariant_id`: 唯一 ID (kind-INV-NNN 格式)
- `describe`: 这条不变量在查什么 (一句话)
- `red_check`: 一份明显错的实例, 验证基线必须不通过
- `green_check`: 一份最简对的实例, 验证基线必须通过
- `gradient`: A/B/C/F 四档实例的描述
- `verify_method`: 具体判定法, 取值 `dom_assertion / pixel_diff_threshold / vision_llm_simple / vision_llm_compare_to_baseline / json_schema / regex_match / ast_walk / 其他`
- `severity_if_fail`: blocker / major / minor / cosmetic 四档

漏填后果: 验证基线退化成形式合规检查 (有样本路径就算过), 失去"修订前后判别力对比"能力, 修订模板可能在没人发现的情况下退化。
检查项:
- [ ] 至少 1 条不变量?
- [ ] 每条都有 red_check / green_check / gradient 四档跟 verify_method 跟 severity_if_fail?
- [ ] verify_method 不是含糊措辞, 是具体判定法?

### 字段: standards_index_path_match

干嘛: 让 `docs/standards/_meta/standards-index.yaml` 跟模板自动同步。模板独立写自己的 path_match 路径, 索引文件按这个值更新。
怎么填: 数组形式, 列这种 kind 实例的所有合法路径 glob, 例如:
```yaml
standards_index_path_match:
  - "src/omnicompany/**/<kind>s/*.py"
  - "src/omnicompany/**/<kind>.py"
```
漏填后果: SemanticAuditor 不知道这种 kind 实例在哪, 守护扫描漏盲区。
检查项:
- [ ] 至少 1 条 path_match?
- [ ] 跟 instance_location.pattern 字段一致 (没有跨 instance_location 之外的路径)?

## 整体检查项 (做完所有四个文件后再走一遍)

- [ ] 四个文件都建了 (`omnicompany/templates/<新 kind>/{范本,骨架,向导,注册件}.{md,yaml}`)
- [ ] 每个文件都有 OmniMark v3 头, type 都是 `template`
- [ ] 范本.md 跟骨架.md 字段顺序完全一致
- [ ] 骨架.md 的 placeholder 都是语义提示
- [ ] 向导.md 一字段一节, 每节四问题
- [ ] 注册件.yaml 的 checks 数量跟规范硬条款对齐
- [ ] 规范文件 `docs/standards/<新 kind>.md` 先立了, 模板没有跑在规范前面
- [ ] 跑 `omni template lint --kind=<新 kind>` 全过 (实装后)
- [ ] 沙盒 `omni sandbox check --content=<示例实例> --kind=<新 kind>` 通过 (实装后)
