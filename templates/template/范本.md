
# 元模板范本 — 一份完整模板长什么样

这份范本展示**一份满分模板长什么样**。读这份的目的: 你将来要给某种 kind 新建模板时, 看完这份就知道"我要做出来的东西长这个样子", 不用再猜。

下面用 **plan** (项目过程记录) 这种 kind 作教学示范 — 假设我们要给 plan 立模板, 它的完整四件套长这样。注意: **这只是示范, 不是真实的 plan 模板**; 真实的 plan 模板是 plan 标准定下来之后立的另一份独立产出。

---

## 一份完整模板的物理结构

如果是 plan 这种 kind 的模板, 落在文件系统里是这样的目录:

```
omnicompany/templates/plan/
  范本.md       — 一份完整 plan 实例的具体例子, 字段全填好
  骨架.md       — 给 plan 实例用的空填表, 字段位置都留好
  向导.md       — 怎么填骨架的指引, 一字段一节
  注册件.yaml   — CLI 用的机器可读规则
```

每份文件都要有自己的 OmniMark v3 头, type=template, agent 字段跟创建 session 关联。模板自己的注册中心登记一条, kind=template, sub-kind=plan。

## 文件 1 / 4: `omnicompany/templates/plan/范本.md` 长什么样

**这是给 plan 实例用的范本** (不是元模板的范本, 是 plan 这种 kind 的模板的范本)。它本身是一份字段全部填好、可以直接被人读懂的 plan 实例例子:

```markdown

# omnicompany 注册中心实装计划

## 主题
按 omnicompany CLI 草案的"实装顺序建议"段, 把身份组+材料组单条+材料组批量+沙盒组+锁组+指引组逐段实装。

## 起止日期
2026-04-15 起 — 2026-05-01 止 (估计 16 天, AI IDE 全职)

## 参与方
- AI IDE 主实施
- 用户审议关键决策点 (锁开关交付时刻 / 批量注册兜底机制 / 等)

## 关联材料
- 上游: docs/standards/cli/omnicompany_cli.md (CLI 草案规范)
- 上游: docs/standards/cli/sandbox.md (沙盒已建, 实装时复用)
- 输出: src/omnicompany/packages/services/omnicompany/cli.py (新建)
- 输出: src/omnicompany/packages/services/omnicompany/registry.py (新建)
- 输出: docs/PROGRESS.md 各阶段更新

## 阶段拆分
1. 身份组 (register identity / whoami) — 2 天
2. 材料组单条 (register material) — 3 天
3. 沙盒组 (sandbox check / promote / archive) — 3 天
4. 材料组批量 (register batch) — 2 天
5. 锁组 (lock open / close / status) — 3 天
6. 指引组 (guide / reflect) — 3 天

## 收尾条件
所有六组 CLI 命令真跑通 + 端到端实例化测试通过 + 锁打开且无误伤 24 小时。

## 风险
- 锁开关时机如果对现有未注册内容覆盖太多, 守护误伤会多
- 批量注册的部分失败兜底, 需要先跑过几次大批量才能调参

## 收尾归档位置
docs/plans/[2026-04-15]OMNICOMPANY-CLI-PHASE3/
```

读这份范本, 任何人立刻能猜出来"哦原来 plan 实例就是这种格式 — 主题 / 起止日期 / 参与方 / 关联材料 / 阶段 / 收尾条件 / 风险 / 归档位置"。这就是"范本"作为材料合规线**之上**的"满分参照"作用。

## 文件 2 / 4: `omnicompany/templates/plan/骨架.md` 长什么样

**这是给 plan 实例用的骨架** (空填表):

```markdown
<!-- [OMNI] tags=plan,<主题领域>,<阶段标签>,<其他你需要的>

# <计划主题, 跟 OmniMark summary 一致或更详细>

## 主题
<2-3 句话讲这份计划要做什么, 跨越的范围, 大致目标>

## 起止日期
<起始日期> 起 — <结束日期> 止 (估计 <X 天/周>, <谁> 全职/兼职)

## 参与方
- <谁主实施>
- <谁审议关键决策点>
- <其他参与方>

## 关联材料
- 上游: <这份计划基于哪些已有材料>
- 输出: <计划做完会产出哪些新材料>

## 阶段拆分
1. <阶段一名称> — <估计天数>
2. <阶段二名称> — <估计天数>
...

## 收尾条件
<什么发生算这份计划完成>

## 风险
<已识别的风险点列表>

## 收尾归档位置
docs/plans/[<起始日期>]<主题代号>/
```

骨架里所有 `<...>` 是 placeholder, placeholder 内描述**这一项要填什么语义内容**, 不是写 `<string>` 这种类型描述。骨架字段顺序跟范本完全一致。

## 文件 3 / 4: `omnicompany/templates/plan/向导.md` 长什么样

**这是给 plan 实例用的向导** (跟骨架配套, 一字段一节):

```markdown
<!-- [OMNI] tags=template,plan,guide

# plan 模板填写指引

## 字段: 主题
干嘛: 让所有读这份计划的人 30 秒内知道这份计划的目标。
怎么填: 2-3 句话, 跨越范围 + 大致目标, 不要堆代号或英文术语。
漏填后果: 后续团队接手时不知道这份计划为什么存在, 容易跟其他计划重复。
检查项:
- [ ] 一个外部读者读完主题段, 能复述这份计划的核心目标吗?
- [ ] 跟跨越范围一致 (不是讲某个具体阶段)?

## 字段: 起止日期
干嘛: 给计划的时间预算和度量基准。
怎么填: 起始用计划真正开始动的那天, 结束是预期完成日 (允许后续修订)。括号里写估计天数和谁全职。
漏填后果: 没有预算就没有"逾期"概念, 计划永远在跑, 最后变成"项目"而不是"计划"。
检查项:
- [ ] 跟当前主线节奏不冲突?
- [ ] 估计天数有依据 (例如阶段拆分加总), 不是拍脑袋?

## 字段: 参与方
干嘛: 让审议环节知道找谁; 让重叠工作的人知道彼此在做什么。
怎么填: 列出主实施者 + 审议者 + 其他显著参与方。每条带角色标注。
漏填后果: 审议时找不到决策人, 决策没人拍板。
检查项:
- [ ] 主实施者明确?
- [ ] 用户在审议环节列入了 (如适用)?

## 字段: 关联材料
干嘛: 让这份计划在材料图里有位置, 不悬空。
怎么填: 上游 (基于哪些已有材料) + 输出 (会产出哪些新材料), 都写材料 ID 或路径。
漏填后果: 计划做完, 产出材料没有 traceback 到这份计划, 注册中心找不到归属。
检查项:
- [ ] 至少 1 项上游 + 至少 1 项输出?
- [ ] 关联材料 ID 真存在 (跑 omni register lookup 能查到)?

## 字段: 阶段拆分
干嘛: 把"起止日期"那个总预算分到具体步骤上, 让进展可追踪。
怎么填: 编号阶段, 每条带阶段名 + 估计天数。3-7 个阶段比较合理 (太多说明颗粒度太细, 太少说明拆分不够)。
漏填后果: 计划看着是一整块, 中途失败时不知道哪一段炸了, 不知道还剩多少。
检查项:
- [ ] 阶段总天数加起来跟"起止日期"括号里的估计天数一致?
- [ ] 每阶段产出物明确 (不是"做完那阶段")?

## 字段: 收尾条件
干嘛: 定义"什么算这份计划完成"的硬标准, 防止计划无限拖。
怎么填: 一两条具体可测的条件 (例如"X 测试通过"+"Y 文档落盘")。
漏填后果: 计划没有终止线, 被无限"还有一点点要补"拉长。
检查项:
- [ ] 收尾条件可测 (有客观标准)?
- [ ] 跟"输出关联材料"对齐 (输出都齐了才能收尾)?

## 字段: 风险
干嘛: 提前说出已识别的不确定性, 让审议者能挑战。
怎么填: 列出 2-5 条已识别风险, 每条简短描述发生时的影响。
漏填后果: 风险隐而不发, 撞上时反应慢。
检查项:
- [ ] 风险跟当前阶段实际情况贴合 (不是抄历史模板)?

## 字段: 收尾归档位置
干嘛: 计划做完后产出物归档到哪。
怎么填: 项目相对路径, 按 docs/plans/[起始日期]主题代号/ 格式。
漏填后果: 计划归档无规则, 后续找历史时翻不到。
检查项:
- [ ] 路径符合文档分布规范 (在 docs/plans/ 下面)?
- [ ] 起始日期跟计划"起止日期"一致?

## 整体检查项 (填完所有字段后再走一遍)

- [ ] 所有 placeholder 都已被替换 (不再有 `<...>` 残留)
- [ ] OmniMark 头五字段齐 (origin/ts/summary/why/tags), is_canonical_v3 通过
- [ ] 文件位置符合 docs/plans/[YYYY-MM-DD]TOPIC/plan.md 规则
- [ ] 上游材料的 ID 在注册中心能查到
- [ ] 阶段拆分天数加总跟起止日期估计天数一致
```

## 文件 4 / 4: `omnicompany/templates/plan/注册件.yaml` 长什么样

**这是给 plan 实例用的 CLI 配置** (机器可读):

```yaml
# [OMNI] origin=ai-ide domain=omnicompany/templates/plan ts=2026-05-01T00:00:00Z type=template status=active agent=ai-ide-bd9cde92
# [OMNI] summary="plan 模板的 CLI 注册件接口"
# [OMNI] why="omnicompany CLI 实例化 plan 时按这份配置走"
# [OMNI] tags=template,plan,registry-config

kind: plan

instance_naming:
  pattern: "plan.md"
  description: "plan 实例文件名固定 plan.md, 跟父目录的日期+主题代号组合作完整标识"

instance_location:
  pattern: "docs/plans/[<start_date>]<topic_code>/plan.md"
  description: "项目相对路径; start_date 是 ISO8601 日期 (YYYY-MM-DD); topic_code 是大写蛇形 (例如 OMNICOMPANY-CLI-PHASE3)"
  variables:
    start_date: "计划起始日期, 跟 plan.md 内字段一致"
    topic_code: "计划主题代号, 跟 plan.md 主题段语义一致, 大写字母+数字+短横"

required_fields:
  omnimark_header:
    - origin
    - ts
    - type   # 必须 == "plan"
    - summary
    - why
    - tags   # 必须含 "plan"
  body_sections:
    - "主题"
    - "起止日期"
    - "参与方"
    - "关联材料"
    - "阶段拆分"
    - "收尾条件"
    - "风险"
    - "收尾归档位置"

checks:
  - id: "plan-001"
    description: "OmniMark 头 type 必须 == 'plan'"
    rule: "header.type == 'plan'"
  - id: "plan-002"
    description: "tags 必须含 'plan'"
    rule: "'plan' in header.tags"
  - id: "plan-003"
    description: "主题段非空且超过 30 字符"
    rule: "len(body.主题.strip()) > 30"
  - id: "plan-004"
    description: "起止日期段含 'YYYY-MM-DD' 格式起始日"
    rule: "regex_match(body.起止日期, r'\\d{4}-\\d{2}-\\d{2}')"
  - id: "plan-005"
    description: "关联材料段至少 1 项上游 + 1 项输出"
    rule: "count_lines_starting_with(body.关联材料, '- 上游:') >= 1 and count_lines_starting_with(body.关联材料, '- 输出:') >= 1"
  - id: "plan-006"
    description: "阶段拆分段至少 3 个阶段"
    rule: "count_numbered_items(body.阶段拆分) >= 3"
  - id: "plan-007"
    description: "文件位置匹配 docs/plans/[YYYY-MM-DD]TOPIC/plan.md 模式"
    rule: "regex_match(file_path, r'docs/plans/\\[\\d{4}-\\d{2}-\\d{2}\\][A-Z0-9-]+/plan\\.md$')"

header_defaults:
  origin: "ai-ide"          # 实例化时 CLI 自动填, 显式传可覆盖
  type: "plan"               # 不可覆盖
  status: "active"           # 实例化默认 active, 后续可改 deprecated
  agent: "<auto-from-session>"   # 实例化时调 current_writer_identity()

related_standards:
  - "docs/standards/concepts/plan.md"    # 详细规范

reference_anti_patterns:
  - "用 plan.md 装跟 plan 无关的散文档案 — 那是 plan 概念被滥用, 应当走 docs/reports/ 或 sandbox/drafts/"
  - "起止日期跨度超过 3 个月 — 那是项目不是计划, 应当拆成多份 plan 或者立 project (如果 project 这种 kind 已经分立)"
  - "收尾条件用'差不多就行'之类模糊措辞 — 必须客观可测"

counterexample_ledger:
  required_for_implementations: true
  ledger_path_pattern: "data/services/<service>/counterexamples/<rule_id>.jsonl"
  reflect_on_count: 1
  degrade_on_count: 3
  rule_trace_required_fields:
    - rule_id
    - rule_version
    - trigger_context
    - rule_output
    - timestamp

invariants:
  - invariant_id: "plan-INV-001"
    describe: "起止日期总天数跟阶段拆分天数加总一致"
    red_check: "起止 16 天但阶段加总只有 8 天 → 不通过"
    green_check: "起止 16 天 + 阶段加总 16 天 → 通过"
    gradient:
      A: "总天数完全相等"
      B: "差 1-2 天 (取整误差)"
      C: "差 3-5 天 (估算粗糙)"
      F: "差 ≥ 6 天 (拆分跟总账脱钩)"
    verify_method: "regex_match + 数值比对"
    severity_if_fail: "major"

standards_index_path_match:
  - "docs/plans/[*]*/plan.md"

three_layer_correspondence:
  user_layer_concept: "计划 (Plan)"
  protocol_layer_primitive: "Format (markdown 类纯数据)"
  skill_step: "§0 调研已有 + §11 落档"
```

## 这份范本本身的"满分"自检

回头看这份范本展示的内容, 它做到了:

**直接看到一份完整模板的物理结构** — 顶部画了目录树, 一眼看出"哦模板就是四个文件加在一起"。

**真实四份文件的具体内容都在** — 不是说"范本.md 应当包含什么", 是真给你看 plan 模板的范本.md 字面长什么样, 字段全填好, 读完能复述。

**满分而不是合格线** — plan 模板的向导.md 给了 8 个字段每个 4 问题加整体检查项, 不是只说"应该有向导"; 注册件.yaml 给了 7 条 checks 不是只说"应该有 checks"。

**跨整个模板系统的一致性** — 四份文件都用 OmniMark v3 头, agent 字段都自动跟 session 关联, 文件名严格按"范本.md / 骨架.md / 向导.md / 注册件.yaml"四件套, 跟 template 标准对齐。

**避免了"提前看后面"** — 这份范本只是把 plan 作教学例子, 不替代真 plan 模板; 真 plan 模板由 plan 标准定下来后另立, 跟这份范本互不耦合。

读这份的人, 将来要给某种 kind 新建模板时, 直接复制 `骨架.md`, 替换 placeholder, 就有了一份合格起点; 想做满分, 就照这份范本的细致度跟一遍。
