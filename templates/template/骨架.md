
# 新模板骨架 — 复制这份, 改 `<...>` 即得新 kind 模板

将来要给某种 kind (例如 worker / agent / hook / data / 任何新概念) 新建模板时, 复制本目录到 `omnicompany/templates/<新 kind>/`, 然后:

1. 把所有 `<...>` placeholder 替换为目标 kind 的具体内容
2. 把 `范本.md` / `骨架.md` / `向导.md` / `注册件.yaml` 四个文件每个都按下面的填空表改

## 1 / 4: 新 kind 的 `范本.md` 应该包含什么 (复制这份骨架填空)

```markdown
<!-- [OMNI] tags=template,<新 kind>,sample,<其他>

# <新 kind> 范本

<这里写一份完整的 <新 kind> 实例的真实例子, 字段全部填好, 没有任何 placeholder>

<示范的具体例子选什么? — 选这种 kind 在系统里真会出现的形态; 不要用 foo/bar 占位, 而是用具体业务里真存在的实体>

<...> 内容应当让一个外部审阅者只看这份范本就能立刻知道这种 kind 的实例长什么样, 不需要查规范 <...>
```

## 2 / 4: 新 kind 的 `骨架.md` 应该包含什么 (复制这份骨架填空)

```markdown
<!-- [OMNI] tags=<新 kind>,<其他你需要的>

<这里给出新 kind 实例的字段集合, 每个字段位置都留好, placeholder 是 <这一项要填什么的语义说明>>

<placeholder 的写法规范:
  - 不写 <string> 这种类型描述
  - 写 <这个字段在这种 kind 里要装什么具体内容>
  - 例如 plan 的"主题"字段 placeholder 是 <2-3 句话讲这份计划要做什么>, 不是 <主题>
>

<字段顺序跟范本.md 一致, 不重排>
```

## 3 / 4: 新 kind 的 `向导.md` 应该包含什么 (复制这份骨架填空)

```markdown
<!-- [OMNI] tags=template,<新 kind>,guide

# <新 kind> 模板填写指引

<对应骨架里的每一个字段, 写一节, 一节回答四个问题>

## 字段: <字段一名称>
干嘛: <一句话讲这个字段的语义角色>
怎么填: <取值范围 / 命名规则 / 引用其他材料的方式>
漏填或填错的后果: <跑不通会怎样, 守护会怎么报警, 下游会出什么问题>
检查项:
- [ ] <自检条件 1>
- [ ] <自检条件 2>

## 字段: <字段二名称>
...

## 整体检查项 (填完所有字段后再走一遍)
- [ ] 所有 placeholder 都已被替换
- [ ] OmniMark 头五字段齐 (origin/ts/summary/why/tags)
- [ ] <跟 <新 kind> 规范的关键对接点检查>
```

## 4 / 4: 新 kind 的 `注册件.yaml` 应该包含什么 (复制这份骨架填空)

```yaml
# [OMNI] origin=ai-ide domain=omnicompany/templates/<新 kind> ts=<创建时间> type=template status=active agent=<创建 session 关联>
# [OMNI] summary="<新 kind> 模板的 CLI 注册件接口"
# [OMNI] why="omnicompany CLI 实例化 <新 kind> 时按这份配置走"
# [OMNI] tags=template,<新 kind>,registry-config

kind: <新 kind>

instance_naming:
  pattern: "<新 kind 实例的文件名规则, 例如 worker_*.py 或固定 plan.md>"
  description: "<这种命名规则的语义解释>"

instance_location:
  pattern: "<实例应该写到哪个目录, 用 <var> 形式让 CLI 替换>"
  description: "<位置规则的语义解释>"
  variables:
    <variable 1>: "<这个变量怎么生成>"
    <variable 2>: "<这个变量怎么生成>"

required_fields:
  omnimark_header:
    - origin
    - ts
    - type   # 必须 == "<新 kind>"
    - summary
    - why
    - tags   # 必须含 "<新 kind>"
  body_sections:    # 如果实例是 markdown 类型, 列出必有章节
    - "<章节一>"
    - "<章节二>"
  body_fields:      # 如果实例是代码 / yaml 类型, 列出必有字段
    - "<字段一>"
    - "<字段二>"

checks:
  - id: "<kind-001>"
    description: "<这条规则在查什么>"
    rule: "<可执行的判定逻辑, 跟 omnicompany CLI lint 引擎对接>"
  - id: "<kind-002>"
    description: "<...>"
    rule: "<...>"

header_defaults:
  origin: "<默认 origin, 通常 ai-ide>"
  type: "<新 kind>"   # 通常不可覆盖
  status: "active"
  agent: "<auto-from-session>"

related_standards:
  - "docs/standards/<新 kind>.md"     # 详细规范文件路径

reference_anti_patterns:
  - "<这种 kind 常见的反模式 1>"
  - "<这种 kind 常见的反模式 2>"

counterexample_ledger:
  required_for_implementations: true                      # 必有, 不允许 false
  ledger_path_pattern: "<反例账本落盘路径模式, 含 <rule_id> 占位>"
  reflect_on_count: 1                                     # 1 条反例触发反思
  degrade_on_count: 3                                     # 3 条反例触发规则降级
  rule_trace_required_fields:
    - rule_id
    - rule_version
    - trigger_context
    - rule_output
    - timestamp

invariants:
  - invariant_id: "<新 kind>-INV-001"
    describe: "<这条不变量在查什么, 一句话>"
    red_check: "<一份明显错的实例, 验证基线必须不通过>"
    green_check: "<一份最简对的实例, 验证基线必须通过>"
    gradient:
      A: "<做得很好的实例特征>"
      B: "<做得不错的实例特征>"
      C: "<勉强通过的实例特征>"
      F: "<不达标的实例特征>"
    verify_method: "<dom_assertion / pixel_diff_threshold / vision_llm_simple / vision_llm_compare_to_baseline / json_schema / regex_match / ast_walk / 其他具体判定法>"
    severity_if_fail: "<blocker / major / minor / cosmetic>"
  # ... 至少 1 条不变量, 复杂 kind 可多条

standards_index_path_match:
  - "<这种 kind 实例的合法路径 glob, 例如 src/omnicompany/**/<kind>s/*.py>"
  # ... 至少 1 条, 跟 instance_location.pattern 一致

three_layer_correspondence:
  user_layer_concept: "<这种 kind 在 omnicompany 用户八种概念里叫什么>"
  protocol_layer_primitive: "<对应六元原语: Hook / Tool / Node / Format / Signal / Intent>"
  skill_step: "<对应 SKILL.md 11 步主步骤>"
```

## 整体检查项 (建完新 kind 模板后再走一遍)

- [ ] 四个文件 (范本.md / 骨架.md / 向导.md / 注册件.yaml) 都建了
- [ ] 四个文件都有 OmniMark v3 头, type 都填正确
- [ ] 范本.md 的字段顺序跟骨架.md 完全一致
- [ ] 范本.md 内容是真实例子 (不是 foo/bar 占位)
- [ ] 骨架.md 的 placeholder 是语义说明, 不是类型描述
- [ ] 向导.md 一字段一节, 每节回答四问题, 不展开成手册
- [ ] 注册件.yaml 的 checks 跟 docs/standards/<新 kind>.md 的硬条款对应
- [ ] 跑 omni template lint --kind=<新 kind> 通过
- [ ] 跟规范文件 docs/standards/<新 kind>.md 已经先立了 (没立的话先补规范, 别先补模板)
