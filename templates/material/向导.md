
# 材料模板填写指引

读这份的前提: 你已经看过 `范本.py` 知道一份合规材料定义文件长什么样, 现在要给某个服务包 / 领域包真做一份 `formats.py`。每一节对应骨架.py 里的一段, 教你怎么填。

## 起手三件事 (动手前必走)

### 命名迁移声明

新建材料定义文件用 omnicompany 当前命名: 类名跟概念叫 Material, 从 `omnicompany.packages.services._core.omnicompany` import (Material 是 protocol 层 Format 的 alias 100% 等价). Registry 类仍用 `FormatRegistry` (旧名 grandfathered, 跟真业务一致, 例如 services/_diagnosis/semantic_auditor/formats.py 同形态). 业务材料的 description / tags / docstring 一律用"材料 / Material"这一新名. 完整对照在 `docs/standards/_global/terminology.md`.

### 公司级材料唯一入口

涉及 plan / progress-entry / project / capture / review-material 五类公司级材料时, 先读 `docs/plans/agent-framework/[2026-06-13]LLM-CALL-UNIFICATION/authority-confirmation.md` 与 `autonomous-execution-rules.md`。这五类统一到 `protocol.Format + FormatRegistry.register`, 不允许在本模板实例里新增平行枚举、平行 registry 或审阅材料私有 kind 权威。

### 第一件: 读规范

权威规范在 `omnicompany/docs/standards/concepts/material.md`, 包含 F-01~F-19 硬规则、FA-01~FA-15 反模式。读完规范再开骨架, 不要边填边查。

### 第二件: 回答三灵魂问题

你准备新建的这份材料, **存起来有什么用**? 三件事至少占一件:

- **能帮诊断吗?** — 出问题时, 这份材料本身能不能成为诊断线索? 在 trace 里看到能不能说出"这一步出错原因可能是 X"?
- **能给人类看和改吗?** — 人类需要时能不能直接打开看懂、改、塞回去?
- **能让计算机运行 / 解析 / 生成吗?** — 下游工人 / 守护 / 检索能不能直接 parse?

三件都不占的, **不应该新开 material** — 藏在产生它的 Worker 内部走 Signal 即可。

### 第三件: 检查跟现有材料的重复度

跑 `omni register lookup --domain=<你的域>` 看现有材料列表。如果你准备做的内容**跟某份现有材料的 (内容 ∩ 操作性) 都一样**, 不要新开 — 直接复用现有的, 或者扩展现有的字段。

但: 人操作 vs 机器操作不算一样。同一份信息的"人改的 .md"和"机器读的 .json"可以独立存在 (访问模式不同)。

## 字段: id

干嘛: 这份材料在整个 omnicompany 注册体系里的唯一标识。
怎么填:
- 用点分层级命名: `<domain>.<stage>.<aspect>` 或 `<domain>.<具体名>`
- domain 用本服务包名 (例如 `csv_to_md`) 或本领域名 (例如 `gameplay_system`)
- 全小写, 短横不下划线
- 例: `csv_to_md.parsed_rows`, `gameplay_system.xlsm-formula-map`, `voxel_engine.entity_spec`

漏填后果: 注册中心冲突 / 下游订阅找不到。
检查项:
- [ ] domain 跟你的服务包目录名一致?
- [ ] 跑 `grep -r "id=\"<你的id>\"" src/` 没冲突?

## 字段: name

干嘛: 人类可读名称, 给 trace-view / dashboard / 错误信息显示用。
怎么填:
- 驼峰命名 (例如 `XlsmFormulaMap`, `ParsedRows`, `MdOutput`)
- 应当完整描述这种材料的语义角色, 不要缩写到看不懂
漏填后果: trace-view 显示的是 id 字符串, 人类阅读体验差。
检查项:
- [ ] 不靠注释能读懂这个名字代表什么类型的数据?

## 字段: description (五要素 — 最关键)

干嘛: 这是协议的"语言锚定"之根。LLM 和人类都靠这一段理解材料。description 不合格 = 材料不合格。
怎么填: 五要素逐项展开 (材料规范 F-01 强制), 用文档块结构清晰呈现:

1. **字段语义** — 每个字段在业务里到底是什么, 不是写"string / int" 类型
2. **值域 / 枚举 / 不变量** — 字段的取值范围、合法值清单、字段间不变式
3. **上游承诺** — 进入本材料时, 上游 Worker 已保证哪些事 (例如 "headers 长度 ≥ 1, rows 每行 len 等于 headers 长度")
4. **下游用途** — 哪个 Worker 消费这份材料, 用了哪些字段, 怎么用
5. **最小合法样例** — 一份完整的具体 JSON 实例, 所有 required 字段填合法值

漏填或敷衍后果:
- 下游 Worker 写代码时只能猜字段语义 → 高概率误用
- 一致性丢失 — 不同人读 description 得出不同理解 → 系统语义漂移
- LLM 节点拿这份材料作输入时不知道字段语义 → 幻觉

检查项:
- [ ] 字段语义按业务讲, 不是按类型讲?
- [ ] 上游承诺有几条 (≥ 2 条)?
- [ ] 下游用途明确点了哪个 Worker 哪个字段怎么用?
- [ ] 最小样例是真 JSON, 字段全填好的?
- [ ] 一个外部审阅者只看 description 能复述材料的结构和语义吗?

## 字段: parent

干嘛: 语义继承链。子类型保持父类型的意图但改变表达结构。
怎么填:
- 选系统 BUILTIN_FORMATS 里已有的根类型作 parent (常见: `requirement` / `doc` / `code` / `material` / `transform`)
- 不存在合适根类型时, 在你自己的服务包内先定义父类型, 再让子类型 parent 指向它
- 不允许悬空 parent (指向不存在的 id)
漏填后果: 类型兼容性检查失效, PipelineChecker 报错。
检查项:
- [ ] parent 在 BUILTIN_FORMATS 或同 package 内已定义?
- [ ] parent 的语义跟本类型继承关系合理?

## 字段: json_schema

干嘛: 给计算机做**协议层结构校验**用; 也帮 LLM 理解 payload 字段类型。
**重要 — material 本体格式跟 json_schema 是两件事**: json_schema 描述的是 Format 协议层的字段约束 (传什么字段、字段类型是什么); material 的本体可以是任何格式 (结构化 JSON / markdown / 代码 / 二进制 / 多文件)。schema 不锁死本体, 本体格式由 description + tags 描述。

怎么填:
- 用标准 JSON Schema (Draft 2020-12) 描述 payload 字段
- 必填字段写 `required` 列表
- 字段类型用 `type` (string/integer/array/object 等)
- 每个字段加 `description` 跟 description 段的字段语义对齐 (不要矛盾, 也不要重复)
- **大明文字段** (markdown / 代码 / SQL 等): schema 标 `type: string`, description 详细说明本体内容结构 (例如"四节固定 markdown 文档")
- **指针字段** (workspace 文件): schema 标 `type: array` 或 `type: object` 描述指针结构 (workspace + relpath + hash + size_bytes), description 说明指针指向什么
- **二进制字段**: 不嵌 payload, 走指针型 (本体走 workspace 文件)

漏填后果: 实例 schema 校验跑不动, LLM 没结构化提示。
检查项:
- [ ] required 列表跟 description 里"required"标的一致?
- [ ] 每个字段的 description 跟 description 段的字段语义一致?
- [ ] 用 jsonschema.validate 试 examples 字段都通过?
- [ ] 大明文字段是 type=string 不是嵌套 schema (本体结构在 description 里讲清楚)?
- [ ] 指针字段含 workspace + relpath + hash + size_bytes 等元数据?

## 字段: material 本体格式

干嘛: 显式回答"这份 material 的本体是什么格式"。material 本体**可以是任何计算机数据** — 结构化 JSON / markdown 文档 / 代码 / 二进制 / 多文件协同 / 甚至不可读的混淆代码。规范 (F-17) 定了大明文阈值, 但具体哪种本体格式适合, 要按 material 的语义角色选。

**两条硬规则**:

1. **schema 必须可读** — 不论本体可不可读, material 的 schema (description + json_schema + tags 三个协议层字段) 必须始终是人和机器都能理解的。schema 是 material 的对外接口, 整个 omnicompany 系统靠 schema 做协议层操作 (校验 / 路由 / 诊断 / 检索), 不依赖本体可读性。
2. **可读性是优选指标** — 同等条件下选可读形态。不可读本体 (二进制 / 加密 / 混淆) 也允许作为 material 传递, 但要在 description 里解释为什么不能可读, 给出可读替代 (反汇编摘要 / 元数据视图)。可读本体的诊断 / 给人看 / 给机器解析三件事都更省力。

material 本体常见的几档形态 (或混合):

### 档 A · 结构化数据型

特征: payload 全是 JSON 标量 / 对象 / 数组, 字段值是结构化的, 机器直接 parse 用。
何时用: material 本体就是结构化数据 (例如解析后的 CSV 表 / xlsm 公式映射 / API 响应)。
示例: `csv_to_md.parsed_rows` (headers + rows 二维数组) / `gameplay_system.xlsm-formula-map` (字段+公式映射)。
schema 写法: 完整 JSON Schema 描述每个字段类型。
tags 加: `body_format.structured_data`。

### 档 B · 大明文嵌入型

特征: payload 含 `<content_field>: str` 字段, 嵌入大段非结构化文本 (markdown 文档 / 代码 / SQL / 等)。
何时用: 本体是大文本, 大小 < 10KB (F-17 阈值), 人类直接读, LLM 用语义抽取。
示例: `gameplay_system.table-understanding-doc` (markdown 字段嵌入正文) / `gameplay_system.table-edit-script` (Python 字段嵌入脚本)。
schema 写法: 大明文字段标 `type: string`, description 详细说明本体内容结构 (例如四节固定 markdown / 函数签名约束)。
tags 加: `body_format.embedded_text`。

### 档 C · 指针型 (F-17 推荐)

特征: payload 含 `files_ref` 列表, 大明文 / 二进制 / 多文件本体走 workspace 文件。
何时用 (材料规范 F-17 硬阈值):
- 单条 ≥ 10 KB: 建议走指针
- 单条 ≥ 1 MB: 强制走指针
- 二进制 / 多媒体: 强制走指针
- 多文件协同 (例如打包工程): 强制走指针
示例: `voxel_engine.built_jar` (jar 二进制 + 编译报告) / 任何图片 / 数据库文件。
schema 写法: `files_ref` 字段是 array, 每项含 workspace / relpath / ts / hash / size_bytes 元数据。
写入规则: 唯一合法入口是 WorkspaceWriterWorker 子类 (工人规范 R-22)。
tags 加: `body_format.workspace_pointer`。

### 档 D · 混合型

特征: 部分字段结构化, 部分字段是大明文嵌入或指针。
何时用: material 本体复杂, 既有协议层元数据 (业务 ID / 版本号 / 时间戳) 又有大明文内容或文件引用。
示例: 多数复杂 sink material (例如 jar 产物 material 同时含 entity_name 元数据 + files_ref 指针 + compile_log_summary 短文本嵌入)。
schema 写法: 各字段按其类型分别处理。
tags 加: `body_format.mixed`。

漏填后果:
- 默认 JSON 思维下, 大文档 / 二进制材料被强行嵌 payload → DB 膨胀 / 查询慢 / replay 成本高 (违反 F-17)
- 指针型材料没用 WorkspaceWriterWorker → 文件落盘审计断链 (违反 R-22)
- 大明文嵌入型不在 description 里说明本体内容结构 → 下游 Worker 不知道 markdown 该有几节 / 代码该有什么签名

检查项:
- [ ] 已经显式选了一档 (A/B/C/D) 加进 tags 的 `body_format.*` 标?
- [ ] 选档 B 的, description 里详细说明本体内容结构?
- [ ] 选档 C 的, schema 含完整指针字段 (workspace + relpath + hash + size_bytes)?
- [ ] 选档 C 的, 产出 Worker 是 WorkspaceWriterWorker 子类?
- [ ] 内容大小跟选档一致 (没把 1 MB 大文档塞档 B)?

## 字段: tags (kind 三分必有 + 业务域 + 内容性质)

干嘛: 多维度分类, 给 Q4 诊断 / dashboard 搜索 / 守护规则消费。
怎么填:

**必有: kind 标 (三选一, F-19 强制)**

| kind | 语义 | 何时用 |
|---|---|---|
| `kind.source` | 系统外部输入, 无 producer Worker | 团队启动信号, 由 CLI / 外部事件 / 定时器注入 |
| `kind.internal` | worker 间流转 | 中间产物, 至少一个上游一个下游 (大部分材料是这档) |
| `kind.sink` | 系统最终输出, 允许无下游 | 落盘 / CLI 响应 / 审计日志 |

**必有: 业务域标** — `domain.<服务包名/领域名>`, 例如 `domain.csv_to_md` / `domain.gameplay_system`。

**按需: 内容性质标** — 例如 `content.tabular_data` / `content.markdown` / `content.xlsm_formula` / `content.code_diff`。

**按需: 来源/稳定性/成熟度标** — 例如 `source.excel-primary` (一手信息源) / `stability.schema-cycle` (schema 稳定) / `status.experimental`。

漏填后果:
- 缺 kind 标 → Q4 诊断误报孤儿 / 冗余, 守护警告刷屏
- 缺 domain → dashboard 按域过滤失效

检查项:
- [ ] tags 至少 2 项 (kind + domain)?
- [ ] kind 在 source / internal / sink 三选一?
- [ ] 没有矛盾 tag (例如同时 kind.source 又 kind.sink)?

## 字段: examples

干嘛: 让 LLM / 新人看到具体长什么样; 让 jsonschema.validate 测试可跑。
怎么填:
- 至少 1 个 example, 所有 required 字段都填合法值
- 多个 example 覆盖不同场景 (例如空集 / 单条 / 多条 / 边界值)
- example 字段值应当跟 description 里"最小合法样例"一致或扩展 (不要矛盾)
漏填后果: F-07 不合规, 学习成本高, schema 校验没测试用例。
检查项:
- [ ] 至少 1 个 example?
- [ ] 跑 jsonschema.validate(example, json_schema) 全过?

## 整体检查项 (填完所有字段后再走一遍)

- [ ] OmniMark 头五字段齐 (origin/ts/summary/why/tags), is_canonical_v3 通过
- [ ] 文件位置在 `src/omnicompany/packages/<services|domains>/<name>/formats.py` 或 子包内 `formats.py`
- [ ] DOMAIN 常量跟服务包目录名一致
- [ ] 每份 Material 的 id 全局唯一 (跑 grep 验证)
- [ ] 每份 Material 的 description 五要素齐
- [ ] 每份 Material 的 tags 含 `kind.*` 之一
- [ ] 每份 Material 的 parent 在 BUILTIN_FORMATS 或同 package 已定义
- [ ] 每份 Material 至少 1 个 example
- [ ] ALL_MATERIALS 列表含全部定义
- [ ] register_formats 函数存在且注册全部 material
- [ ] 跑 `python -m py_compile <你的 formats.py>` 通过
- [ ] 跑 `omni sandbox check --content=<你的 formats.py> --kind=material` 通过
- [ ] 跟 docs/standards/concepts/material.md 的 F-01~F-19 逐条对齐 (尤其 F-01 五要素 / F-15 声明即消费 / F-19 kind 必有)
- [ ] 三层对应声明: 这份材料对应协议层 Format 原语 (纯数据, 不调 LLM 不改状态不做决策), 对应 SKILL 实施层 §2 Format 体系跟 §3.1 节点设计单表的 format_in / format_out 字段
