
# Dashboard Team 可观测性界面规范

本规范约束 dashboard 里 Team 图、TeamBuilder 实战归因、doctor/debug 轨迹、material 观察相关界面。

## 核心原则

1. 固定显示文本必须优先使用中文。英文 ID、代码符号、文件路径可以保留英文，但必须配中文代称、中文说明或人读摘要。
2. 不要把底层日志字段直接当成主界面。`srcomnicompany...`、原始 bash 命令、JSON、源码片段只能作为调试证据，不得作为人类审阅的第一入口。
3. Team 观察界面的第一目标是回答“发生了什么、为什么这么判断、我该点哪里核对”，不是展示数据库里有哪些字段。
4. worker、material、生成产物、读取线索、已确认读取必须有不同视觉语义，不能长得完全一样。
5. 所有图上可审阅对象都应在当前界面打开小窗、悬浮窗或侧栏详情。新窗口只允许作为“原始记录/源码调试入口”，不能是主阅读路径。

## 归因层命名

1. `material_io_links` 展示为“声明输入/输出”。含义是 Team/Worker 定义里写明应该读写什么。
2. `produced_content_materials` 展示为“生成产物”。含义是外部代理本次实际产出的文件或内容。
3. `resource_material_links` 展示为“实战读取线索”。含义是运行记录显示 worker 接触过某个文件、目录、命令、查询或外部资源，但尚未确认它等价于正式 material。
4. `inferred_material_read_links` 展示为“已推断读取”或“文件声明命中”。含义是系统已经根据 material_id、声明头或明确 token 把读取线索升级成 material 关系。
5. 不使用“工具读取候选”“按定义读取”这类需要解释才能理解的词作为 UI 主标签。

## 详情卡要求

每个归因节点的详情至少包含：

1. 人读判断：一句话说明这条证据代表什么。
2. 判断依据：说明系统为什么这样判断，引用工具事件、文件声明、路径命中或 material_id 命中。
3. 具体目标：文件路径、目录模式、命令摘要或查询目标。
4. 资料摘要：如果目标是本地文件，展示文件类型、路径、大小、可见标题或可见定义；不要默认展开大段源码。
5. 置信度和状态：说明这是声明事实、生成产物、待确认线索，还是已确认 material 读取。
6. 原始记录入口：只能作为调试链接，标签必须明确写“原始记录”或“原始 JSON”。

## 聚合节点要求

资源空间这类聚合节点必须显示具体例子：

1. 卡片上展示关联 worker 数量、线索数量和至少一个具体例子。
2. 点击聚合节点必须打开详情，列出若干具体线索和关联 worker。
3. 聚合节点必须明确说明“这是资料入口线索，不是已确认 material 边”。

## 图面交互要求

1. 右侧栏、底部事件流、悬浮详情等辅助区域必须可关闭或可恢复。
2. 图节点尺寸必须稳定，文本不能裁剪核心信息；长 ID 放到次级行或详情里。
3. 可拖动节点必须保持可用，布局算法可以提供初始布局，但不能阻止人工整理。
4. 引入布局或连线库时优先使用现有前端栈内的轻量库，不为单一视图引入过重服务。
5. 参考成熟流水线、DAG、trace viewer 的表达方式，但最终要符合 omnicompany 的 material/worker/team 语义。

## AI 摘要要求

当原始证据是 bash、grep、read_file、源码或 JSON 时，dashboard 不应要求人直接阅读原文。优先由内部 agent、Claude Code worker 或本地启发式摘要生成短报告：

1. 读了什么资料。
2. 为什么认为它和某个 worker/material 有关。
3. 置信度和待确认点。
4. 可点击的原始证据链接。

短报告是主产物，原始证据是可追溯材料。

## 验证要求

改动 Team 可观测性界面时至少验证：

1. `npx tsc -b`
2. 相关 Playwright e2e
3. 8210 页面实机探针：打开 TeamBuilder team，点击 worker、material、生成产物、实战读取线索、资源空间聚合节点，确认都有当前界面详情。
4. 固定显示文本检查：不得新增主要英文标签或需要人解释的内部术语。
