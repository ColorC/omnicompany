# 模板标准

## 一、 概念三层分类 (omnicompany 概念体系的根)

omnicompany 项目里所有概念按职责跟落地位置分三层. 模板系统按这三层分别提供模板, 不是平铺八种 kind:

### 层一 · AI 控制体系 (src 内, 跑业务的)

src 内部跑 AI 控制系统的六种核心概念:

- **agent** — 含 LLM 调用 + 工具调用 + AgentNodeLoop 动态探索的 worker 子类型
- **hook** — 接外部信号产 source material, 不调 LLM 不改状态不做决策
- **tool** — 执行确定性操作 (读 / 写 / 跑命令 / 调 API), 不调 LLM 不做决策
- **material** — Format 协议层的纯数据, worker 间通信介质
- **team** — worker 拓扑组合, 含 pipeline.py + 多个 worker.py + DESIGN.md + .omni/manifest.yaml 等多文件结构
- **worker** — Node 协议层, 业务逻辑执行者

这层概念在协议层对应六元原语 (Hook / Tool / Node / Format), 落到 `src/omnicompany/packages/services/<service>/` 跟 `src/omnicompany/packages/domains/<domain>/`. 模板放在 `templates/<kind>/`, 范本是文件夹结构 (因为真实例就是多文件).

### 层二 · 文档体系 (src 外, 跟人类交互的)

跟人类交互 / 给人读 / 让人审议的两种文档:

- **data** — material 落盘后的物理形态, 含 sidecar v1.1 元数据. 落到 `data/services/` / `data/domains/` / `data/_workspaces/` 等
- **doc** — 人读的散文档, 含三种子类:
  - **plan** — 跨日跨阶段过程记录, 落 `docs/plans/<package_path>/[date]TOPIC/plan.md` (按本规范定义的目录结构)
  - **report** — 跨阶段总结报告, 落 `docs/reports/<category>/`
  - **DESIGN** — 服务包级别的设计文档, 归 team 一部分 (不独立成 kind), 落 `src/omnicompany/packages/<service>/DESIGN.md`

DESIGN 在物理位置上就近 team (在 service 包内), 但语义上是"team 的设计文档". 修订 team 时一并修订 DESIGN.

### 层三 · 元模板 (产生上面两层的母模具)

- **template** — 元概念, 用来定义"上面两层每种 kind 的模板长什么样" 的模板. 自身递归三级停 (元模板 → kind 模板 → 实例)

### 三层对应表 (跟协议层 + SKILL 实施层)

| 层 | omnicompany 概念 | 协议层原语 | SKILL 实施层 |
|---|---|---|---|
| 一 (AI 控制体系) | agent | Node (含 LLM 子类型) | §1 / §3 / §3.4 / §6 |
| 一 | hook | Hook (只观测) | §1 / §5 |
| 一 | tool | Tool (只执行) | §1 / §6 |
| 一 | material | Format (纯数据) | §2 / §3.1 |
| 一 | team | Node 拓扑组合 | §4 |
| 一 | worker | Node | §1 / §3 (19 项) |
| 二 (文档体系) | data | Format 落盘 + sidecar | §2 落盘表达 / P-14~P-17 |
| 二 | plan (doc 子类) | Format (markdown) | §0 调研 + §11 落档 |
| 二 | report (doc 子类) | Format (markdown) | §11 落档 |
| 二 | DESIGN (team 一部分) | Format (markdown) | 跟 team 同生命周期 |

不在用户层概念里但协议层有的两个原语 (Signal / Intent), 不立顶级模板:
- **Signal** — 节点内部传值用, 跨调用不持久, 不需要顶级模板
- **Intent** — 用户意图描述, 落到 doc / agent system_prompt 等具体材料里, 不另立基础概念

## 二、 模板是什么

模板是一份**用来产出某一类材料实例**的成品规约。它装四样东西:

- **范本** — 一份字段全部填好、可以直接被人读懂的完整例子 (复杂 kind 是文件夹结构, 含真实多文件)
- **骨架** — 一份字段位置都留好、值未填的填写表
- **向导** — 跟骨架配套的填写指引, 每个字段为什么填、怎么填、漏填的后果
- **注册件接口** — 实例化时 omnicompany CLI 怎么调用、校验时跑哪些规则、注册中心要登记哪些信息

**模板本身就是材料**。它有自己的文件头、自己的命名规则、自己的位置规则、自己的生命周期, 跟任何一份普通材料一样在系统里被注册和管理。它不是悬在系统外面的"参考资料"。

## 二、 模板的核心价值

模板的存在不是为了"装四样东西的形式合规", 是为了**让使用者真正能理解和做出合格的实例**。具体作用四件:

### 作用 1: 骨架让使用者快速建立标准化内容

直接复制骨架就有了合格起点。不用每次新建实例时重新想"应该有哪些字段、字段顺序怎样、必填什么"。骨架是"已经走过一次坑的人沉淀下来的最小起点"。

### 作用 2: 链接到规范和现存代码, 帮助使用者理解内容

模板里必须带两类链接:

- **规范链接** — 链回 `docs/standards/<kind>.md`, 让使用者从模板返回去查权威定义
- **代码链接** — 链回现存代码里这种 kind 的真实合格实例 (具体路径), 让使用者直接读现成的"看代码就懂"

只有规范链接不够。规范是文字描述, 抽象概念使用者读完仍然不知道"具体长什么样"。代码链接给的是真东西, 一眼看穿。

### 作用 3: checklist 和向导防止使用者犯错

向导一字段一节回答四问题, 注册件含可执行的 checks 列表, 整体检查项填完后再走一遍。这三层让使用者在不同阶段都能自检, 不让漏字段、不让填错、不让位置不对、不让头不齐这些常见错误漏过去。

### 作用 4: 用案例诠释抽象概念, 让"道理清楚但不知道怎么做"变成"具体长这样"

这是模板最重要的作用, 也是规范无法替代的。规范说"按价值拆分"、"单一语义责任"、"先需求产物不按接线" — 这些都是抽象原则, 使用者读完知道"道理对", 但不知道"我手上这份具体内容到底应该怎么按这些原则做"。

模板必须**用现存代码里的具体案例诠释每条抽象原则**。例如材料模板里讲"按价值拆分"时, 不能只复述规范, 要列:

> "人类可读价值的体现": 现存材料 `demogame.table-understanding-doc` 之所以独立成 material, 是因为团队 lead 要在 trace-view 里一眼看出表的语义 — 链接 `<具体路径>`
> "调试价值的体现": 现存材料 `voxelcraft.compile_report` 之所以独立, 是因为 jar 编译失败时不光要知道"失败了", 还要看 javac 的具体错误行 — 链接 `<具体路径>`
> "反例 (不该拆的)": 某中间产物只在节点内部存在, 对外无任何调试/复用/人类可读价值 — 这种应当藏在节点里走 Signal, 不开新 material

这才是"诠释抽象概念"的正确形态 — 不再是抽象解释, 是真把现存代码里的具体例子+链接+解释为什么这样体现某个抽象原则。读完使用者能立刻把抽象原则映射到自己手上的内容上。

## 三、 设计材料 / 工人 / 团队等抽象 kind 时要回答的灵魂问题

这一节是给材料模板、工人模板、团队模板等含大量抽象概念的 kind 模板**专门**设的一节, 不适用于纯形式 kind (例如 plan / hook 这种字段相对固定的)。

设置一份新材料 / 新工人 / 新团队时, 系统会把它存起来 / 注册起来 / 跑起来, 但**存起来有什么用**? 灵魂问题三条:

### 能帮诊断吗?

这份材料 (或工人产出 / 团队跑) 出问题时, 它本身能不能成为诊断线索? 在 trace 里看到这份材料能不能说出"哦, 这一步的输入是这样, 所以下一步出错的原因可能是 X"? 如果答案是"看不出来", 这份材料的字段设计就缺了诊断维度。

### 能给人类看和改吗?

人类需要这份材料时 (例如调试 / 审议 / 修订), 能不能直接打开看懂? 看完能不能直接改? 改完能不能塞回去? 如果材料是机器编码格式 (例如打包成 binary 或 schema 复杂的嵌套 JSON), 人类介入成本就高, 应该考虑拆出一份给人看的版本。

### 能让计算机运行 / 解析 / 生成吗?

机器需要这份材料时 (例如下游工人消费 / 守护扫描 / 检索召回), 能不能直接 parse? schema 能不能用 jsonschema validate? 字段语义能不能用 LLM 自动抽取? 如果材料是纯散文 markdown 没结构化字段, 机器消费成本就高, 应该考虑加结构化层。

### 多维度价值评估

回答完三个灵魂问题, 一份材料 / 工人 / 团队的设计可以从五个维度评估:

- **人类易用性**: 人能看懂、能改、能用?
- **人类工作流程中的存在性 (和已有工作可对比性)**: 这份材料对应人类已有的工作物吗? (例如配表对应已有的 xlsx, 设计文档对应已有的 .md) 让人能"啊这就是我以前的 X", 而不是要从零理解一个新概念
- **内部内容语义一致性**: 这一份内部多个字段是不是描述同一件事? 还是混了不同语义? (例如把 schema + diff + 规则证据三种语义混进同一份就违反一致性)
- **和其他 material 的重复度**: 内容和操作性都一样就没必要新开。但人操作 vs 机器操作不一样 — 人改的 .md 跟机器读的 .json 即使内容能映射, 也可以独立存在 (人和机器的访问模式不同)
- **计算机可操作度**: 计算机能 parse / 运行 / 检查这份材料吗?

模板的"案例诠释抽象概念"那一节, 应当用现存材料 / 工人 / 团队作具体例子, 一条一条说清楚这五个维度怎么体现。

## 四、 跟其他概念的边界

### 模板与材料

模板是材料的一种, 用 `kind=template` 区分。任何材料标准里讲的字段、命名、注册、生命周期约束, 模板都一样要守。

模板跟"模板的目标材料"不是同一份东西。一份 `template.worker` 模板说的是"工人这种材料应该长什么样", 不是某个具体工人。一份具体工人是按 `template.worker` 实例化产出的另一份独立材料。

### 模板与规范

规范 (`docs/standards/<kind>.md`) 描述某种材料**应该满足什么**, 是文字描述。
模板 (`omnicompany/templates/<kind>/`) 描述某种材料**长什么样**, 是可以直接复制和填写的成品。

规范是判断依据, 模板是产出工具。两者必须一致 — 模板的范本不能违反对应规范, 否则系统出现"按模板做就对, 按规范做就错"的矛盾。模板修订必须先改规范、再同步模板; 反过来不允许。

### 模板与实例

一份模板对应零到多份实例。实例是按模板新建的、字段已经填好的具体材料。

实例不是模板的复制 — 实例填空时可能补充模板没有的字段 (在向导允许的范围内), 也可能产生跟模板范本完全不同的具体值。模板只约束"骨架结构 + 必填字段 + 命名规则 + 位置规则", 不约束实例的具体内容。

### 模板与沙盒里的 guides

沙盒目录的 `guides/` 子目录里放的是**规范副本**, 不是模板。规范副本是从 `docs/standards/` 拷过来的只读复制, 让沙盒里的写入者不切目录就能查规范。

模板本体在 `omnicompany/templates/<kind>/`, 跟规范并列。沙盒的 `guides/` 不存模板本体, 但可以软链或拷贝过来作便利。

### 模板与 omnicompany CLI

CLI 的 `omni new <kind>` 命令实例化模板。CLI 读模板的注册件接口决定要校验什么、要塞什么文件头、要往哪个目录写。模板修订后 CLI 自动用新模板, 不需要重启或者同步配置。

### 模板与 DESIGN.md / manifest.yaml / 反例账本

立一种新概念时, 模板四件套不是唯一辅助产出物, 还有几份配套设施级文件应当一并落:

- **DESIGN.md** (放在 `src/omnicompany/packages/<服务>/DESIGN.md`): 七节硬结构 (状态 / 核心目的 / 核心接口 / 架构决策 / 数据流 / 已知局限 / 参考资料)。模板讲"长什么样", DESIGN.md 讲"这个具体服务的设计是什么样"。两者各管一面。
- **`.omni/manifest.yaml`** (放在 `src/omnicompany/packages/<服务>/.omni/manifest.yaml`): 服务包的依赖契约和材料清单。模板实例化产出真材料后, manifest 把这些材料登记到服务包的"输入/输出/中间"清单。
- **反例账本 `data/services/<服务>/counterexamples/<rule_id>.jsonl`**: 死规则的在线监测账本。每份模板的注册件里 `checks` 段产出的检查规则, 一旦触发就要落账; 出 1 条反例触发反思, ≥ 3 条触发规则降级。这是死规则的安全网, 不是可选项。

立新概念时的工作清单: 先立规范 → 再立模板 → 实例化时同步建 DESIGN.md 跟 manifest.yaml → 模板的 checks 跑起来后开反例账本。前后顺序不要颠倒。

### 概念三层对应 (回查根定义见 §一)

任何一份具体 kind 的模板, 概念定义那一节必须显式给出"这种 kind 在三层分类里的位置 + 协议层原语 + SKILL 实施层步骤". 完整对应表在本规范 §一 三层分类节. 否则模板跟协议层脱钩, 实例化产出的实例不知道自己是什么原语, 守护跟下游消费方都不知道边界.

## 五、 模板的字段

每份模板包含九项内容, 都是必有的, 不允许省略其中任何一项。

前七项是结构性字段 (实例化时被消费), 后两项 (代码案例链接 / 抽象概念诠释) 是**核心价值字段** — 没有这两项, 模板就退化成"形式合规的空壳", 失去了"诠释抽象概念让使用者真懂"的核心作用。

### 元信息块

模板自己的 OmniMark 头, 跟任何材料一样。`type=template`, `domain=omnicompany/templates/<目标 kind>`, `summary` 一句话讲这份模板是给哪种材料用的, `why` 讲这份模板存在的语义理由 (为什么需要这种概念有专门的模板), `tags` 至少含 `template` 和目标 kind 名 (例如 `template,worker,foundation`)。

### 概念定义

跟模板配套的"这种材料是什么"的一句到三句话定义。这一节不是规范的复制, 是给打开模板的人一个最快速的"这是干嘛的"提示。详细定义回查规范。

### 范本

至少一份完整的真实例子。"完整"意思是所有字段都填好, 没有任何 placeholder。"真实"意思是字段值是这种材料在系统里真会出现的形态, 不是 `foo` / `bar` 这种占位词。

如果这种材料有显著不同的子类型 (例如工人有简单大语言模型工人 / 不带大语言模型工人 / 智能体三档), 范本应当一档一份, 不混在一起。

范本应当是 "如果一个外部审阅者只看这份范本, 不看任何其他文档, 就能立刻知道这种材料应该长什么样" 的水平。这是范本的硬质量门槛。

### 骨架

跟范本结构对应的空填写表。所有需要填的字段位置都留好, 值是 `<...>` 形式的 placeholder, placeholder 里写**这个字段要填什么语义内容**, 不是写字段类型 (例如不是 `<string>` 而是 `<这个工人接收的输入材料的 ID>`)。

骨架的字段顺序应当跟范本完全一致。没有可选字段的概念 — 实例化时所有骨架字段都必须被填, 哪怕填"无"或者"暂无"。

### 向导

跟骨架配套的填写指引, 一字段一节, 每节回答四个问题:

1. **这个字段是干嘛的** — 一句话语义
2. **怎么填** — 取值范围 / 命名规则 / 引用其他材料的方式
3. **不填或填错的后果** — 跑不通会怎样, 守护会怎么报警, 下游会出什么问题
4. **完成检查项** — 一两条自检清单 (例如"已经在注册中心登记了对应的输入材料 ID 吗?")

向导的总体长度跟模板覆盖的字段数挂钩 — 字段越多向导越长, 不要为了短而省字段说明。但单个字段的解释不要超过半页, 超了说明这个字段太复杂, 应当拆。

### 注册件接口

一份机器可读 (YAML) 的描述, 给 omnicompany CLI 用。包含:

- **实例命名规则**: 实例文件名应该长什么样 (正则或者描述性规则)
- **实例位置规则**: 实例应该放到哪个目录 (项目相对路径, 含 `<>` placeholder 让 CLI 替换)
- **必填字段清单**: 跟骨架对齐, CLI 校验时用
- **校验规则 (`checks`)**: 实例要通过哪些检查才算合规 (例如"工人必须声明 FORMAT_IN 和 FORMAT_OUT", 引用对应规范条款)
- **OmniMark 头默认值**: 实例化时自动填什么 origin / type / tags 等
- **反例账本配置 (`counterexample_ledger`)**: 实例化产出的实例里, 任何死规则触发都要走反例账本机制。注册件里声明账本路径模式 (例如 `data/services/<服务>/counterexamples/<rule_id>.jsonl`)、触发反思的反例阈值 (默认 1 条触发反思 / 3 条触发规则降级)、必有的 rule_trace 字段集 (rule_id / rule_version / trigger_context / rule_output / timestamp)。这一项是**必有**, 不允许空着 — 死规则没有反例账本就是裸奔
- **验证不变量 (`invariants`)**: 实例的红绿测试基线不只列样本路径, 还要列每条不变量的 `red_check` (绝对错应不通过) / `green_check` (最简对应通过) / `gradient` (做得 A/B/C/F 四档区分) / `verify_method` (dom_assertion / pixel_diff / vision_llm_simple / vision_llm_compare 等具体判定法) / `severity_if_fail` (blocker / major / minor / cosmetic)。验证不带不变量声明就是没分辨力的验证
- **关联标准 (`related_standards`)**: 列对应规范文件路径, 让模板修订时同步知道改哪份规范, 让 SemanticAuditor 能从模板回查规范
- **standards-index 同步检查**: 注册件应当声明自己在 `docs/standards/_meta/standards-index.yaml` 里的 path_match 项, 让索引文件跟模板自动一致

### 反模式列表

跟这种材料绑死的常见反模式 (从规范复制, 让模板内自检也能跑)。这一节是为了让模板的使用者**不用切到规范文档**就能立刻知道哪些做法是错的。

### 代码案例链接 (核心价值字段)

每份模板必须列出**至少 5 份现存代码里这种 kind 的真实合格实例**, 给项目相对路径或绝对路径作链接。

为什么是 5 份: 1-2 份不够覆盖变体差异, 看的人会以为某种特例就是通例; 5 份能跨越主要使用场景, 让看的人通过对比真实例子建立"这种 kind 的常态分布感"。

每条链接要带:
- **路径**: 项目相对路径 (例如 `src/omnicompany/packages/services/csv_to_md/workers/extract_worker.py`)
- **诠释维度**: 这份案例主要诠释了什么 (例如"诠释了 worker 的 R-19 智能体子类型" / "诠释了 material 的 F-15 声明即消费")
- **看点提示**: 一句话告诉读者打开这份代码后该看哪一段 (例如"看 ContextScript / LLM / Tool 三件套的 if 分支, 那是 R-19 的字面体现")

代码案例链接是**活的** — 链接的代码被废弃 / 重构 / 迁移时, 模板要同步更新链接。模板修订时这一项是必查项, 不能让链接腐烂成"指向不存在的路径"。

### 抽象概念诠释 (核心价值字段)

模板必须**显式列出对应规范里的抽象原则**, 每条原则用至少一份案例真诠释。

什么算"抽象原则": 规范里那些"应该满足 X" 但 X 本身不是机器可校验、需要人类判断的条款。例如:

- 材料规范的"按价值拆分": 价值有四类 (人类可读 / 调试 / 复用 / 检查薄弱性), 哪类多重才该拆? 规范不给数字门槛, 只能用案例诠释
- 工人规范的"单一语义责任": 一件语义还是两件? 用案例对比"应当合并的两步"和"应当拆分的两步"才能讲清
- 团队规范的"先需求产物不按接线": 规范说反对接线思维但什么样的设计算"先需求产物"? 用一个真实团队的演化过程作案例
- 智能体规范的"工具自由度高 + 三位一体": 引导/规范/安全网三层边界怎么把握? 用 business_explorer 7 工具反例 vs bash + 必要包装的正例对比

诠释的形式 (硬规则):

> **抽象原则名** (来自规范第 X 条 / 链接到具体段落)
> **规范原话** (一句到三句话引用)
> **正面案例 N 份** (各自附路径 + 一段话讲为什么这份代码是这条原则的字面体现)
> **反面案例至少 1 份** (附路径 + 一段话讲为什么这份代码违反这条原则, 应当怎么改)
> **判断准则** (从案例对比里抽出来的, 给将来面临类似选择的使用者: "如果你手上的内容含 X 特征, 倾向走 A 路径; 含 Y 特征, 倾向走 B 路径")

每条抽象原则的诠释**长度不限**, 因为这一节就是给抽象概念落地用的, 关键是真把案例搞透。短的诠释 (一两句) 没用; 真长的诠释 (跨几个案例对比 + 判断准则) 才有价值。

不是所有 kind 都需要这一节: 形式相对固定的 kind (例如 plan / hook 这种字段相对枚举的) 可以只有 1-2 条抽象概念诠释或者干脆这一节空着 (但要在注册件.yaml 显式声明 `abstract_concepts_explained: []` 表示已审视过没需要诠释的抽象点)。

含大量抽象概念的 kind (材料 / 工人 / 团队 / 智能体) 这一节必须丰满, 否则模板就是空壳, 实例化产出的实例会因为使用者读不懂抽象原则而频繁违规。

## 六、 命名规范

模板的目录命名: `<kind>/` 在 `omnicompany/templates/` 下面。`kind` 用单数英文 (因为目录名要稳定, 不跟着语言切换), 跟材料 kind 系统对齐 (`worker` / `material` / `team` / `agent` / `hook` / `tool` / `data` / `plan` / `template`)。

注意 `template/` 也是模板系统的一部分 — 模板自己的模板 (元模板)。这是个递归点, 三级以上递归没意义, 元模板就是模板的模板就停。

模板内部文件命名:
- `范本.<目标扩展>` (例如 `范本.py` 给工人模板, `范本.md` 给计划模板)
- `骨架.<目标扩展>`
- `向导.md`
- `注册件.yaml`

如果一份模板有子类型分档变体 (例如工人有三档), 用子目录:
- `<kind>/<变体>/范本.<扩展>` (例如 `worker/simple_llm/范本.py`)
- `<kind>/<变体>/骨架.<扩展>`
- `<kind>/<变体>/向导.md`
- `<kind>/<变体>/注册件.yaml`

不允许在文件名里挂版本号 (`范本_v2.py` 是禁的)。版本通过 OmniMark 头的 `ts` + git history 管。

### 命名迁移 (新名优先)

新建模板跟新立实例都按 omnicompany 当前命名: 材料 (Material) / 工人 (Worker) / 团队 (Team) / 仓库 (Stock) / 部门 (Department) / OmniCompany. 旧名 (Format / Router / Pipeline / EventBus / 项目顶层 omnicompany) 仅在协议层旧 import 路径里保留, 业务层模板和实例都用新名。`docs/standards/_global/terminology.md` 记完整对照。模板范本里 `from omnicompany.protocol.format import Format` 这种是协议层旧名 grandfathered, 不传染业务层。

## 七、 位置规范

所有模板落在 `omnicompany/templates/` 下面, 跟规范 (`docs/standards/`) 平行。理由是模板是规范的"产出工具版", 在文档区里跟规范同根最合理。不放代码区 — 模板不是代码, 实例化时被 CLI 复制到代码区, 模板本身不参与运行时。

不允许把模板放在沙盒、归档区、或者各服务包内部。沙盒的 `guides/` 是规范副本不是模板; 各服务包内部要新概念时, 仍走 `omnicompany/templates/<kind>/` 找模板, 不允许服务包自己造模板。

如果系统将来需要某种领域特化模板 (例如 demogame 业务有自己的工人变体), 在 `omnicompany/templates/<kind>/<领域>/` 子目录里加, 不另起根目录。

## 八、 实例化协议

实例化是从模板产出一份具体实例的标准流程。CLI 走这套协议, AI IDE 和用户也走同样协议 (不允许绕过)。

### 第一步: 选定目标 kind 和子变体 (如有)

调用方告诉 CLI 要做哪种概念的实例。`omni new worker --variant=simple_llm` 选择工人的简单大语言模型变体。CLI 在 `omnicompany/templates/worker/simple_llm/` 找模板。

### 第二步: 复制骨架到目标位置

CLI 读 `骨架.<扩展>` 和 `注册件.yaml` 的位置规则, 把骨架复制到目标位置 (例如 `src/omnicompany/packages/services/foo/workers/bar_worker.py`)。骨架文件里所有 placeholder 都还在 (留给填空)。

CLI 在骨架顶部插入 OmniMark 头 (按注册件接口规定的默认值生成, summary / why / tags 暂时填占位字符串等填空时被覆盖)。

### 第三步: 提示填空 + 显示向导

CLI 把骨架文件路径返回给调用方, 同时把 `向导.md` 内容显示出来 (或者打开编辑器同时显示两份)。调用方按向导填骨架的 placeholder。

向导建议的填写顺序:从结构性字段 (id / 输入材料 / 输出材料) 开始, 到行为性字段 (run 方法 / 校验逻辑) 结束。这样早期发现结构错误省后期返工。

### 第四步: 自检 + 注册中心校验

填空完成后, 调用方调 `omni sandbox check --content=<填好的实例路径> --kind=<kind>`, CLI 按 `注册件.yaml` 的校验规则跑全套检查。检查项包括:

- 所有必填字段是否填了 (没有残留的 `<...>` placeholder)
- 字段值是否符合命名规则
- 文件位置是否符合位置规则
- 引用其他材料的字段 (例如 FORMAT_IN) 是否真存在
- 文件头五字段 (type / origin / summary / why / tags) 是否齐全
- 跟规范条款的对接点是否都过

不通过时给具体哪一条没过, 调用方修了再跑。通过才能进下一步。

### 第五步: 注册到中心 + 转正式区

调用方调 `omni register material --kind=<kind> --path=<目标路径>` 拿写入凭据, 然后或者调 `omni write` 让 CLI 写, 或者调用方自己写但带凭据。

CLI 把这一份实例登记到注册中心 (在 `<workspace>/.omni/registrations/` 下面记一条), 关联回模板 (用了哪份模板的哪个变体), 关联 session ID 和身份 token。

注册成功之后这份实例就是合规的"在册材料", 防修改锁开启后受保护、被巡检覆盖。

### 实例化时的工作区隔离

实例化产出的实例如果会引发后续运行 (例如新建一个工人, 后续会跑这个工人产数据), 调用方应当走工作区隔离 — 每次 run 用 git worktree / cp -r / 容器作独立工作区, 禁硬编码路径常量, 避免跨 run 污染。这条铁律在 `docs/standards/_global/workspace_isolation_and_test_discrimination.md` 第一条立, 任何模板的实例化都应当遵守。模板的注册件接口可以在 `instance_runtime` 字段声明这种 kind 实例的工作区隔离要求。

## 九、 模板的生命周期

### 创建

新增一种概念时, 由 AI IDE 或用户人手写一份模板。流程跟普通材料注册一样 — 在沙盒 `drafts/` 写出范本+骨架+向导+注册件, 自检 (`omni sandbox check --kind=template`) 通过后转到 `omnicompany/templates/<kind>/`, 注册中心登记。

新增模板必须同时新增对应规范 (`docs/standards/<kind>.md`)。规范先, 模板后。如果规范不存在直接想做模板, 要么先补规范, 要么这个 kind 还不该立。

### 修订

模板的字段、范本、向导、注册件随系统进化需要修订。修订流程:

1. 先改对应规范 (`docs/standards/<kind>.md`), 让规范变化先稳定
2. 再改模板的对应部分 (范本 / 骨架 / 向导 / 注册件)
3. 跑 `omni template lint <kind>` 自检模板自身是不是合规
4. 注册中心更新模板的 OmniMark `ts` 字段标记修订时间
5. 已经按旧模板产出的实例**不自动迁移**, 留给后续的 "实例升级" 阶段统一处理

允许小修小补 (例如向导里加一句话、修个错别字) 不算修订, 直接改即可。结构性变化 (字段增减、必填变可选、校验规则修改) 才算修订。

### 实例升级

模板修订后, 旧实例可能不达新模板的标准。处理这种情况的两条路:

- **被动升级**: 守护巡检发现旧实例不合规时报警, 由人或 agent 在沙盒里改一份合规版本走 promote 替换。适合修订小、影响小的场景。
- **主动升级**: omnicompany CLI 提供 `omni template upgrade --kind=<kind>` 命令, 扫所有用旧模板生成的实例, 按新模板自动更新可以自动更新的字段 (例如新增可选字段填默认值), 不能自动的留给人。适合修订大、影响广的场景。

主动升级是 omnicompany CLI 阶段三之后的功能, 当前不强制实装, 但应当在 CLI 规范里留口子。

### 验证不变量三档判别力

修订模板时, 升级后的模板要带得动**有判别力的验证基线**, 不只是有红绿样本。判别力三层 (按 `docs/standards/_global/workspace_isolation_and_test_discrimination.md` 跟 `docs/standards/_global/verification_invariants.md`):

- **绝对门 (red_check)**: 一份明显错的实例, 验证基线必须不通过. 通过了 = 验证基线没分辨力, 修订无效
- **绝对门 (green_check)**: 一份最简对的实例, 验证基线必须通过. 不通过 = 验证基线过严或者错位
- **梯度区分 (gradient A/B/C/F)**: 一系列做得"很好 / 不错 / 勉强 / 不达标"的实例, 验证基线给出有差异的反馈. 全 PASS 或全 FAIL = 没梯度

模板修订前后都要跑判别力自检 — 修订前的旧模板有判别力, 修订后的新模板必须至少同等判别力, 否则修订就是退化。这条检查应当固化在 `omni template lint` 命令里。

### 归档

某种概念被弃用时 (例如某种工人变体不再需要), 模板进入归档状态。流程:

1. 修改模板的 OmniMark 头 `status=deprecated`, 加一行注释指向替代方案
2. 模板**不立即删**, 保留在 `omnicompany/templates/<kind>/<变体>/` 让历史实例还能查
3. CLI 的 `omni new` 不再为弃用变体提供实例化, 但 `omni template show` 仍能展示
4. 归档保留期由系统配置决定 (推荐至少一年, 因为已有实例可能还在跑)
5. 保留期过后, 模板和所有引用它的实例 (如还有) 一起进入归档区

### 版本管理

模板没有显式版本号字段。每次修订只改 OmniMark `ts` 字段标记修订时间。"版本"通过 git history 完整可追溯。

如果某次修订对外行为产生不兼容变化, 应当拆成新的变体 (在 `omnicompany/templates/<kind>/<新变体>/`), 而不是把旧变体改坏。两个变体并存, 由调用方在 `omni new --variant=` 选。

## 十、 反模式

下面列举模板设计或实例化中确认会引发后续问题的做法。这一节是给写模板和审模板的人作 checklist 的, 不是给实例化使用者看的 (实例化使用者看自己 kind 的反模式列表即可)。

### 反模式: 模板里塞具体业务字段

模板应当是 kind 通用的, 跨业务可复用。如果工人模板的范本带着 demogame 业务的具体字段名 (`SeasonBookHonor`, `TavernCreationPool` 之类), 这份模板就锁死在 demogame, 不能给 voxelcraft / narrative 用。

正确做法: 范本用通用占位 (`Foo` / `Bar` / `Baz`) 但语义明确 (`InputDocument` / `OutputSummary`); 子变体可以贴近某种使用场景但不带业务实体名。

### 反模式: 范本是过期代码

范本必须是当下系统里**真会出现**的形态。如果范本里有 `from omnicompany.runtime.deprecated_module import *` 这种过期 import, 实例化出来的实例从一出生就是历史包袱。

正确做法: 修订模板时, 范本一并审, 跟现存代码风格保持一致。每年至少一次模板范本审计 (走 omnicompany agent 跑批量, 不走 AI IDE 手做)。

### 反模式: 只有范本没有骨架

骨架是给"会填空但不知道字段哪里有 placeholder"的使用者用的。光有范本时, 使用者读完不知道哪些字段是必须自己填、哪些是可以照抄, 容易把范本里的具体值当成模板默认值带进实例。

正确做法: 范本和骨架并列存在, 骨架的 placeholder 跟范本的具体值一一对应, 让使用者一眼看出"哪里是我必须填的"。

### 反模式: 向导写成手册不是清单

向导的目的是辅助填空, 不是教学。如果向导每个字段都展开成几页解释、加上历史背景、加上反模式枚举、加上跨概念跳转, 使用者读完已经忘了自己要填什么。

正确做法: 向导每个字段一节, 一节回答四个问题 (干嘛/怎么填/不填后果/检查项), 总长度在使用者能一次读完的范围内 (经验值: 普通 kind 的向导不超过 800 字)。详细背景去规范里查, 不重复在向导里。

### 反模式: 模板自己没有文件头

模板自己也是材料, 自己也要有 OmniMark 头。否则违反"一切材料化"的根本要求, 模板系统自身就不合规。

正确做法: 模板的范本 / 骨架 / 向导 / 注册件四个文件每份都有 OmniMark 头, 头里 `type=template` 标识自己是模板而不是被它产出的实例。

### 反模式: 用文件名挂版本号

`worker_template_v2.md` 这种命名违反"版本不进文件名"铁律。两个变体并存时该建子目录, 不该建版本后缀的文件。

正确做法: 子目录区分变体, OmniMark 头 `ts` 字段记修订时间, git history 看完整变更。

### 反模式: 模板修订时不改规范

模板和规范必须一致。如果改模板时绕过规范 (因为改规范要审而改模板不需要), 久了模板和规范会漂移, 出现"按模板做就对, 按规范做就错"的矛盾。

正确做法: 修订流程硬规则 — 改规范在前, 改模板在后, 反过来不允许。CLI 的 `omni template lint` 应当包含"模板跟规范一致性"检查作硬门槛。

### 反模式: 模板覆盖太广或太窄

太广: 一份模板想覆盖所有可能的工人形态 (简单 / 复杂 / 智能体 / 特殊变种), 字段无限多, 大部分实例填一半字段空着。
太窄: 每个具体业务的工人都做一份模板, 模板数量爆炸, 维护不动。

正确做法: 模板按"对外行为有显著区别 + 跨业务复用"拆变体。工人变体三档 (简单大语言模型 / 不带大语言模型 / 智能体) 是合理粒度, 因为这三档的字段集差异大且各自跨业务复用。

### 反模式: 注册件接口跟实际 CLI 行为脱钩

注册件接口里写"必填字段是 A, B, C", 但 CLI 实际只检查 A — 实例化产出的实例 B 和 C 字段总是空, 系统出现"模板说要填的, 实际没人填"的失效状态。

正确做法: 注册件接口和 CLI 实装严格绑定, 任何接口字段变化必须同步 CLI 实装代码, CLI 自检模板 (`omni template lint`) 时把"实装是否覆盖所有声明字段"作硬检查。

## 十一、 元模板 (满分参照)

这份标准定义的"模板", 自己也有一份"模板的模板"作元模板存在, 已经真做出来落在 `omnicompany/templates/template/`:

```
omnicompany/templates/template/
  范本.md       — 一份完整模板长什么样的具体示范, 用 plan 作教学例子
  骨架.md       — 给将来新建一种 kind 时复制用的空填表
  向导.md       — 跟骨架配套的填写指引, 一字段一节
  注册件.yaml   — 给 omnicompany CLI 实例化新 kind 模板时用的配置
```

元模板四件套自身合规 OmniMark v3, type=template, agent 字段跟创建 session 关联。

**这份元模板就是"满分模板长什么样"的活参照** — 不是抽象描述什么是满分, 是真把四件套做出来让人看。新建任何一种 kind 的模板时, 应当对照元模板的细致度做, 不是只达"合格线"。

元模板的存在是模板系统自洽的保证 — 模板系统包含产出模板自己的能力, 不依赖任何外部"模板的模板生产机制"。元模板递归三级停 (元模板 → 模板 → 实例), 不做"元元模板"。

## 十二、 跟现存设施的接通点

文件头层 (OmniMark v3): 模板的四个文件 (范本/骨架/向导/注册件) 都用 v3 头, type=template, agent 字段跟当前 session 关联。

身份层 (session 持久化): 实例化模板时, CLI 调 `current_writer_identity()` 拿写入者 ID, 写到实例的 OmniMark 头里。模板自身被修订时同样关联 session。

沙盒层: 新写模板要先在沙盒 `drafts/` 起草, 走 `omni sandbox check --kind=template` 自检 (元模板会被引用作检查依据), 通过后 promote 到 `omnicompany/templates/<kind>/`。

罚单层: 发现绕过模板系统直接产出材料 (例如手写一份不走 `omni new` 的工人) 时, 守护按对应 kind 规范判断是否合规; 不合规走 TOMBSTONE 路径, 没收到 `.omni/quarantine/`, 在原位提醒"要走模板系统"。

注册中心: 每份模板有自己的注册项 (kind=template, sub-kind=worker/material/etc 表示是给哪种实例用的); 每份按模板生成的实例的注册项里关联回它用了哪份模板的哪个变体的哪个 ts 版本。

CLI 命令: `omni new <kind>` (实例化) / `omni template list` (列模板) / `omni template show <kind>` (展示) / `omni template lint <kind>` (自检) / `omni template upgrade --kind=<kind>` (主动实例升级, 阶段三后)。
