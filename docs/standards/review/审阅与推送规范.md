# 审阅与推送规范 · 2026-05-29

> 来源: 用户 2026-05-29 在 BOSS SIGHT 验收时立。用户原话:
> "信息传达的不完整之一包括信息过载, 你推给了我整个网页, 相当于将对照原案看网页的工作推给了我……
> 我发现在审阅的时候不明确我的需求到底实现在哪儿, 不明确什么做了, 什么没做完,
> 不明确理论上你希望我用什么路径去体验什么。"

## 一句话

**推给用户审阅的任何东西, 必须是 curated(经裁剪导览的), 不是 raw(整坨原始内容)。** 把整个网页/整份产物一甩 = 信息过载 = 一种**不完整沟通**——等于把"对照原始需求逐项核对"的活甩回给用户。

## 为什么

`from agent to agent` 的核心是 **agent 把该看的推到人面前**, 而不是把"找该看什么"的工作留给人。一个没有导览的推送, 让用户面对三个"不明确":

1. 不明确**需求实现在哪儿**(这产物对应我哪条需求?)
2. 不明确**什么做了 / 什么没做完**(完成度?)
3. 不明确**该怎么体验**(我应该走什么路径去看什么?)

只要有这三个"不明确", 这次推送就是失败的, 无论产物本身做得多好。

## 规范本体: 每个 reviewable 必带"导览三件套"

任何进入审阅台 / 推送给用户的 reviewable(material 五类: 图 / 文档 / 网页 / 关键问题 / 自定义模板), 推送时必须附带:

| # | 字段 | 含义 | 反例(违规) |
|---|---|---|---|
| 1 | **对应需求** | 这产物满足哪条原始需求 / 用户故事(给出处, 如 `用户故事_增强版.md US-4.x` 或原始需求 §x) | "这是个网页"(没说对应啥) |
| 2 | **完成度** | 什么做了、什么没做完、什么是已知缺陷;一句话状态(完成 / 部分 / 阻塞) | 推一个含测试夹具/半成品的页面却不标 |
| 3 | **体验路径** | 我希望你走什么路径、看什么、点什么、预期看到什么 | 甩一个 URL 让用户自己摸 |

### curation 原则(怎么裁)

- **指向具体物, 不指向整页。** 推"material X 的某一处", 不推"整个 dashboard 自己翻"。
- **替用户做完对照。** "这对应你需求 Y, 已做到 Z, 还差 W"——核对工作 agent 先做完, 把结论给人, 而不是把核对甩给人。
- **空就说空。** 没有真产物时, 显式说"暂无可审产物", 不要推一个装着测试夹具/陈旧数据的页面冒充。
- **分级不抢。** 配合分级中断(必验收顶部条 / 重要角标 / 过程性累计), 别一次性把所有东西都顶到用户面前。

## 适用 / 谁读

- **总控 / 任何 agent**: push material / 提交审阅 / 汇报前, 按本规范自检"导览三件套"齐不齐, 不齐不推。
- **审阅台后端(reviewstage)**: material 应自带这三项元数据字段(对应需求 / 完成度 / 体验路径), 让前端能直接渲染导览, 而不是靠人脑补。这是后续 material schema 的增强方向(不新增 kind, 是给 material 加元数据)。

## 模板

每个 reviewable 的导览元数据按 [templates/reviewable.导览.md](templates/reviewable.导览.md) 填写。material 提交 / push 时把该模板的三段作为元数据带上。

## 维护

- 本规范是 BOSS SIGHT 审阅范式的一部分;后续若新增 reviewable 类型, 同样适用"导览三件套"。
