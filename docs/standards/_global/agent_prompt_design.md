# Agent Prompt 设计规范

LLM agent 的提示词写作准则. 适用于**所有喂给 agent 的 prompt** — AgentNodeLoop 的 NODE_PROMPT、Claude Code / Codex 会话的 system prompt、spawn 给 subagent 的首条指令, 跨域跨业务通用.

## 一 · 核心原则

agent 拿到的 prompt 是它**唯一的工作上下文** — 没读过你的设计文档, 没参与过任何对话, 不知道任何代号, 不知道项目以前是什么样. 写 prompt 要假设读者是一个能力不错但**完全陌生、刚到岗**的工程师. 客观陈述工作目标 + 工具用法 + 文件位置 + 必要案例, 让它自己规划执行细节.

**精简 ≠ 砍能力**: 下面的反模式砍的是"怎么拼这条命令 / 历史交代 / 重复"这类**冗余细节**, 不是"有这个能力"的信号. 最小必要信息的前提是**必要信息要全** — agent 真需要的能力 / 工具 / 约束一个都不能漏. 写完对照真实工具面 (如 `omni <组> --help` 的子命令列表), 确认该会的能力 prompt 都提到了: 既不抄细节, 也不漏能力.

## 二 · 7 类反模式 (硬规则, 写 prompt 时硬避)

### 反模式 1 · 内部代号污染

agent 没读过你的项目控制结构, 不知道层级标号 (L1/L2/L3) / 规范编号 (§2.2 / §3.7) / 规则代号 (R-23 / F-19 / OMNI-014) / 团队叙事词 (铁律 / 用户立). 看到这些只能当成不可解读的字符串.

✅ **正模式**: 直接陈述事情本身, 不用代号引用.
- ❌ "按 SOP §2.2 章节齐"
- ✅ "章节齐 (一句话定位 / 设计目的 / 核心机制与模块 / 资产引用 4 节必有)"
- ❌ "待 L1 仲裁"
- ✅ "待用户仲裁"
- ❌ "用户铁律: sink ≠ 真源"
- ✅ "sink 是摘要, 真源是原文. 你 cat 真源, 不只读 sink"

### 反模式 2 · 冗余命令式语言

"**核心铁律**" / "**必跑**" / "**严禁**" / "**绝禁**" / "**一定要**" / "**应该**" / "**尽量**" 这种元叙事词增加 prompt 体积, 不增加信息. agent 看到 "核心铁律 X" 跟 "X" 收到的信号一样.

✅ **正模式**: 简洁陈述. 必做的事直接命令式, 不加修饰.
- ❌ "**严禁**进入'修 → verify → 再 FAIL → 再修' 死循环. **绝对**修两次不通过就接受 ⚠️ finish, **不要硬刚**."
- ✅ "verify 失败两次接受 ⚠️ finish, 不要循环修."

### 反模式 3 · 过分细致的具体数字

具体数字 (轮数 / 张数 / 字数 / 条数) 在不同业务复杂度下偏差极大. 月卡 5 张图够, 通行证可能要 15 张; 简单业务 10K 字够, 大业务要 50K. prompt 里写死数字会**限死大业务发挥** + **诱导小业务硬凑**.

✅ **正模式**: 给原则不给数字, 让 agent 按材料判断.
- ❌ "选 4-8 张关键图嵌入"
- ✅ "选关键图嵌入 (主功能界面 / 弹窗 / 跨业务跳转), 量按材料丰富度判"
- ❌ "footnote 数 ≥ 30 (大业务 ≥ 50)"
- ✅ "每个数值断言带 footnote, 数量按内容多少自然产生"
- ❌ "通常 4-8 轮够 (helper 1 + cat 1-3 + note 1-2 + finish 1)"
- ✅ "做完工作就 finish, 不限轮数"

**例外 — 真硬上限可写**:
- 安全相关 (例 "单图 ≤ 4 MB, 单轮累计 ≤ 12 MB" — 这是工具协议物理限制)
- 协议契约 (例 schema required 字段)

### 反模式 4 · 极端反例锁死发挥

写 prompt 时为防错列大量"❌ 错: ..." 反例, 反而让 agent 把正例之外的所有变形当作错. 实际上 agent 看到 5 个反例后会**只敢生成 prompt 里出现过的形态**, 失去自主判断.

✅ **正模式**: 给 1-2 个**正模板**让 agent 学结构, 留发挥空间. 反例最多 1 个用于解释抽象原则.
- ❌ 列 "❌ 错: ## P4 配表 / ## collab platform wiki / ## figma" + "❌ 错: ## UE 蓝图轮换 / ## Lua 脚本结构" + "❌ 错: ## 设计演化: 2022年 / 2023年" 三组反例
- ✅ "章节按业务维度分 (例 '## 等级体系 / ## 任务系统 / ## 购买链路'), 而不是按数据来源 ('collab platform' / 'figma') 或工程实现 ('UE 蓝图' / 'Lua') 分."

### 反模式 5 · 开发记录 / 历史叙事

agent 没经历过项目演化, 不知道"以前长什么样""哪天谁提的". prompt 里交代历史变更纯属噪音, 还会误导 — 让它以为有个"旧形态"要兼容, 或去找一份它看不到的旧文档. 典型: 日期署名 ("用户 2026-06-06")、"旧 X 形态 / 本形态不适用"、"已有命令"、"(同 Y)"、"现在改成…".

✅ **正模式**: 只写当前事实, 像 prompt 从来就长这样.
- ❌ "没有 submit_response —— 那是旧 AgentNodeLoop 形态的协议, 本形态不适用"
- ✅ "用自然语言回复结束本轮"
- ❌ "omni plan show <id>   # 已有命令, 看 plan 详情"
- ✅ "omni plan show <id>   # 看 plan 详情"

### 反模式 6 · 同一约束说多遍

一条规则在 prompt 多处复述 (常见: "职责"正面说一遍、"边界"反面再说一遍; 一个约束在三个小节各出现一次) 增加体积, 改时易漏改出不一致.

✅ **正模式**: 每条约束**只在一处**权威落地, 其余引用而非复述.
- ❌ 「职责: 不改代码」+「边界: 不改代码 (subagent 的活)」+「禁写: 一切 .py」三处说同一件事
- ✅ 一处说清 "代码归 subagent, 你不写", 其余不再列.

### 反模式 7 · 抄录自描述面 (CLI --help / 代码真源)

把 `--help` 的全部 flag、枚举值, 或代码里的路径白名单 / 字段表整段抄进 prompt = 给一份会过时的副本 (工具一改 prompt 就错, 实测常已漂移、漏列真实子命令). 这跟"给规范路径让 agent cat"(见第四节) 同理: 自描述的东西让 agent 现查.

✅ **正模式**: 给意图 + 入口, 让 agent 自己查权威面.
- ❌ 列出 `omni review submit --kind ... --tier ... --title ... --plan-id ...` 全部 flag + "kind 五选一: image/markdown/..."
- ✅ "整理产出提交审阅台用 `omni review submit` (参数 / 取值跑 `omni review --help`)"
- 取信顺序: prompt 与 `--help` / 真源不一致时, 以 `--help` / 真源为准.

## 三 · 正模式骨架 (推荐 prompt 结构)

```markdown
# 你是谁
[一句话身份 + 任务边界]

# 工作目标
[要产出什么. 不写步骤, 写终态]

# 可用工具 + 用法
[列工具名 + 1 个 JSON 调用范例 / 工具]

# 资料位置
[真源文件 / 规范文件 / 范本文件 的绝对路径]

# 范本对照
[1 个完整产出范本 + 1 个反范本 (如果有真不该这么做的形态)]

# 收敛判据
[什么情况算完成 (软判据 + 硬判据)]
[什么情况自决 finish 接受不完美]
```

## 四 · 引用规范文件的正确方式

很多 prompt 想表达 "请按规范 X 做". 错误做法是把规范条款编号塞进 prompt (反模式 1). 正确做法是**给规范文件路径让 agent cat 读**:

```markdown
# 第一步 (必读)
cat <规范完整路径>
cat <范本完整路径>
```

agent 自己 cat 后获得规范全文, 不需要 prompt 里复述. 规范变了不需要同步 prompt.

## 五 · checklist 的正确用法

audit / reviewer 类 agent 的 checklist 是**有力的检查形式** — 强制每项明确判 yes/no, 不允许"基本/大致/部分" 含糊. 这跟反模式 3 (枚举不可枚举) 不冲突, 因为:

- checklist 列**类别** (frontmatter 必填字段 / 章节齐 / 嵌图实存 / footnote 闭环 等)
- 类别下的具体条目可以是固定的 (因为这些是协议层强约束)
- critical 违规的判据应该收敛到少数硬条件, 模糊条款转 ⚠️ note 不算 critical

例: 28 项 checklist 全填 yes/no, 但只有其中 5-6 类是 critical, 其余转 note. 这是合理用法.

## 六 · 自检清单 (prompt 写完后 grep 检查)

写完一个新 prompt, 应该 grep 验证无以下污染:

```bash
grep -nE "\bL[12345]\b" prompt            # 反1 内部层级标号
grep -nE "OMNI-[0-9]" prompt               # 反1 内部规则代号
grep -nE "§[0-9]+" prompt                  # 反1 规范条款编号 (除非真用户可见的 RFC/SOP 编号)
grep -nE "铁律|严禁|绝禁|必须" prompt        # 反2 元叙事修饰词
grep -nE "≥[0-9]+|≤[0-9]+|[0-9]+-[0-9]+ 轮" prompt   # 反3 具体数字约束 (除非协议物理上限)
grep -nE "[0-9]{4}-[0-9]{2}-[0-9]{2}" prompt   # 反5 日期署名 (开发记录)
grep -nE "旧.*形态|本形态|已有命令|\(同 " prompt   # 反5 历史叙事
```

任一类有命中, 走相应反模式的修法.

反 6 (重复) 靠通读: 同一约束出现在不止一处就合并到一处. 反 7 (抄自描述面) + 能力完整靠对账: 把 prompt 写到的命令逐条与 `<工具> --help` 对一遍 —— 写死的 flag/枚举删掉换成"跑 --help", 漏列的真实能力补上.

## 七 · 常见疑问

**Q: 不给数字 agent 会不会写少?**
A: agent 看材料丰富度自己判. 真有材料但没产出, 是判据不清而不是数字不清. 改成"产出对应业务复杂度的足够内容" 比 "至少 30 条 footnote" 更准确.

**Q: 不给反例 agent 会不会犯错?**
A: 反例堆多了 agent 反而困惑 (容易把"正确"的变形也当错). 给 1 个正模板 + 简短解释为什么这样是对的, 让 agent 类比迁移. 实际跑出错再增量修.

**Q: prompt 不应该长?**
A: 长不是问题, 信息密度低才是. 一段 prompt 如果删掉所有元叙事词后内容显著缩短, 说明大部分是冗余. 30K 字符的 prompt 可能只有 5K 字符是真信息.

## 八 · 验证方式

prompt 修改前后跑同一业务对照:
- 产出长度 / 章节细分 / 引用密度 / 嵌图数 (内容质量)
- agent 内部代号污染数 (`grep -E "L[12345]|§[0-9]"` 在产出里)
- 收敛 turn 数 (效率)

新 prompt 产出**不污染** + **质量持平或提升** + **效率不降** = 修正成功.
